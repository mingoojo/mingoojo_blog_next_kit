### 1. 시스템 인터페이스 요구사항 분석

1.  **시스템 인터페이스 요구사항 구성**

    > 독립적으로 운영되는 시스템들 사이의 원활한 데이터 교환을 위해 정의해야 할 규칙
    1. 핵심내용
       - 시스템 인터페이스 요구사항은 개발을 목표로 하는 시스템과 외부 시스템을 연동하는데 필요한 시스템 인터페이스에 대한 요구사항을 기술
       - 시스템 인터페이스 요구사항 명세서에는 인터페이스 이름, 연계 대상 시스템, 연계 범위 및 내용 연계 방식, 송신 데이터, 인터페이스 주기, 기타 고려사항 등이 포함
    2. 요구사항 명세서
       - 요구사항 명세서는 프로젝트 개발 시 기업이나 업체가 요구하는 사항들을 구체화하여 명세화한 문서로, 시스템 기능 데이터 인터페이스, 품질 등의 요구사항 단위별로 작성
         <ImageWithCaption src="/static/images/req_statement.png" alt="requestment statement" caption="오구사항 명세서" />

2.  **시스템 인터페이스 요구사항 분석**
    - 시스템 간 연동을 위한 명세서를 꼼꼼히 검토하여 무엇(기능)을, 어떻게(비기능) 할지 구체화하는 과정
    1. 요구사항 명세서의 필수 항목
       - 인터페이스 이름: 식별을 위한 고유 명칭
       - 연계 대상 시스템: 데이터를 주고받는 상대방 (송신/수신)
       - 연계 범위 및 내용: 구체적으로 어떤 업무를 연동할 것인가
       - 연계 방식: 연결 기술 (예: DB Link, Web Service, API 등)
       - 송신 데이터: 주고받는 데이터의 상세 (형식, 길이 등)
       - 인터페이스 주기: 데이터 전송 빈도 (예: 실시간, 매일 밤 12시 등)
       - 기타 고려사항: 보안 설정, 통신 실패 시 오류 처리 등
    2. 요구사항 분석의 목표
       - 분류: 기능적 요구사항(데이터 전송 등)과 비기능적 요구사항(보안, 성능 등)으로 나누기
       - 분해: 요구사항이 너무 덩어리가 크면 알맞게 쪼개기
       - 보충: 누락된 제약 조건이나 빠진 내용 찾아내서 추가하기
       - 우선순위: 어떤 연동이 더 중요한지 등급 매기기
    3. 요구사항 분석의 절차
       - 선별 (Selection): 전체 요구사항 중 인터페이스 관련 항목만 골라 목록 만들기
       - 자료 준비 (Preparation): 아키텍처 정의서, 내/외 연계 현황 등 참고 자료 수집
       - 분류 (Classification): 명세서를 기능적 요구사항과 비기능적 요구사항으로 구분
       - 비교 및 수정 (Comparison): 자료들을 비교하여 누락된 것을 추가하거나 수정하여 구체화
       - 전달 (Delivery): 최종 수정된 명세서와 목록을 이해관계자에게 공유

### 2. 인터페이스 요구사항 검증

1.  **요구사항 검증**
    > 인터페이스 요구사항 검증은 "우리가 앞 단계에서 분석하고 정리한 인터페이스 명세서가 정말 완벽한가?"를 따져보는 과정
    - 요구사항 명세서의 오류가 발견되어 수정될 경우 발생될 비용을 줄이기 위함
    - 요구사항 검토 계획수립 > 검토 및 오류 수정 > 베이스라인 설정의 절차를 거침
    1. 요구사항 검토 계획수립
       1. 검토 기준 및 방법 수립 : 프로젝트의 규모와 참여 인력, 검토 기간 등을 고려하여 검토 기준 및 방법을 선정
       2. 참여자 선정 : 프로젝트 관리자, 품질 관리자, 인터페이스 분석가, 소프트웨어 아키텍트, 시스템 사용자, 테스트 관리자 등 요구사항 검토 참여자를 선정
       3. 체크리스트 작성 : 완전성, 일관성, 명확성, 기능성, 검증 가능성, 추적 가능성, 실현 가능성 등의 항목을 점검할 수 있는 요구사항 검토 체크리스트를 작성
       4. 관련자료 준비 : 인터페이스 요구사항 목록, 인터페이스 요구사항 명세서, 현형 및 표준 시스템 구성도 등 인터페이스 요구사항 검토에 필요한 자료들을 준비
       5. 일정 검토 : 인터페이스 요구사항 검토 일정 선정
    2. 검토 및 오류 수정
       - 체크리스트의 항목에 따라 인터페이스 요구사항 명세서를 검토하는 것
       - 검토 및 수정의 과정
         1. 요구사항 검토 시 오류가 발견되면 오류를 수정할 수 있도록 오류 목록과 시정 조치서를 작성
         2. 오류 수정과 요구사항 승인 절차를 진행할 수 있도록 요구사항 검토 결과를 검토 관련자들에게 전달
         3. 시정 조치서를 작성한 경우 시정 조치가 완료되었는지 확인하여 시정 조치가 완료되면 인터페이스 요구사항 검토 작업을 완료
       - 검증 방법
         1. 동료검토 : 요구사항 명세서 작성자가 명세서 내용을 직접 설명하고 동료들이 이를 들으면서 결함을 발견하는 형태의 검토 방법
         2. 워크스루 : 검토회의 전에 요구사항 명세서를 미리 배포하여 사전 검토한 후에 짧은 검토 회의를 통해 결함을 발견하는 형태의 검토 방법
         3. 인스펙션 : 요구사항 명세서 작성자를 제외한 다른 검토 전문가들이 요구사항 명세서를 확인하면서 결함을 발견하는 형태의 검토 방법
       - 검증 항목
         1. 완전성(Completeness): 사용자의 모든 요구사항이 누락되지 않고 완전하게 반영되어 있는가?
         2. 일관성(Consistency): 요구사항이 모순되거나 충돌되는 점 없이 일관성을 유지하고 있는가?
         3. 명확성(Unambiguity): 모든 참여자가 요구사항을 명확히 이해할 수 있는가?
         4. 기능성(Functionality): 요구사항이 '어떻게(How to)' 보다 '무엇을(What)'에 중점을 두고 있는가?
         5. 검증 가능성 (Verifiability): 요구사항이 실제로 구현되었는지 테스트할 수 있는가?
         6. 추적 가능성 (Traceability): 요구사항이 어디서 왔는지(상위 요구사항) 추적할 수 있는가?
         7. 실현 가능성 (Feasibility): 현재의 기술력과 예산, 일정 안에서 구현 가능한가?
         8. 변경 용이성(Easily Changeable): 요구사항 명세서의 변경이 쉽도록 작성되었는가?
    3. 베이스라인 설정 (승인 후 최종 확정선 결정 과정)
       - 요구사항 검토를 통해 검증된 인터페이스 요구사항은 프로젝트 관리자와 주요 의사결정자에게 공식적으로 승인을 받는 과정
       - 소프트웨어 설계 및 구현을 위해 요구사항 명세서의 베이스라인(작업의 기준점)을 설정

### 3. 인터페이스 방법 명세화

1.  **인터페이스 방법 명세화의 개념**
    > 인터페이스 방법 명세화는 내·외부 시스템이 연계하여 작동할 때 인터페이스별 송·수신 방법, 송·수신 데이터, 오류 식별 및 처리 방안에 대한 내용을 문서로 명확하게 정리하는 것
    1. 인터페이스 방법 명세화 작성 내용
       1. 송수신 방법 명세화
          - 인터페이스의 겉모양과 통로를 설계하는 단계
          - 시스템 연계 기술, 인터페이스통신 유형, 인터페이스 처리 유형, 발생 주기에 대한 내용을 기술
          1. 시스템 연계 기술
             - 개발할 시스템과 내·외부 시스템을 연계할 때 사용되는 기술을 의미
             - 외부 시스템과 통신하는 방법을 정하는 단계로, DB Link, API, Socket 등의 통신법을 정하는 과정
          2. 인터페이스 통신 유형
             - 송신 시스템이 데이터를 보낸 후, 수신 시스템의 응답을 어떤 리듬으로 기다릴 것인지 정하는 단계
             - 동기, 비동기, 단방향 등 통신의 흐름을 정하는 과정이다.
          3. 인터페이스 처리 유형
             - 송·수신 데이터를 어떤 형태로 처리할 것인지에 대한 방식을 정하는 단계
             - 업무의 성격과 송·수신 데이터 전송량을 고려하여 실시간 지연 처리, 배치 방식 등으로 구분
               - 실시간 처리 : 요청한 내용을 바로 처리
               - 지연 처리 : 데이터를 처리량이 많아 대량의 비용이 많이 발생할 때 사용 하는 방식
               - 배치 처리 방식 : 대량의 데이터를 처리할 때 사용하는 방식
          4. 인터페이스 발생 주기
             - 개발할 시스템과 내·외부 시스템 간 송·수신 데이터가 전송되어 인터페이스가 사용되는 주기를 정하는 단계
             - 인터페이스 발생 주기는 업무의 성격과 송·수신 데이터 전송량을 고려하여 매일, 수시, 주 1회 등으로 구분
       2. 송수신 데이터 명세화
          - 실제로 주고받는 내용물이 무엇인지 적는 단계
          - 인터페이서에서 송수신한 데이터의 대한 명세
       3. 오류 식별 및 처리 방안 명세화
          - 인터페이스 시 발생할 수 있는 오류를 식별하고 오류 처리 방안에 대한 명세를 작성
          - 시스템 및 전송 오류, 연계 프로그램 등에서 정의한 예외 상황 등 대·내외 시스템 연계 시 발생할 수 있는 다양한 오류 상황을 식별하고 분류
          - 오류 상황에 대해 오류 코드, 오류 메시지, 오류 설명, 해결 방법 등을 명세화

### 3. 미들웨어 솔루션 명세

1.  **미들웨어(Middleware)의 개념**

    > 미들웨어는 'Middle' + 'Software'의 합성어로, 운영체제와 응용 프로그램 사이에서 매개체 역할을 하는 소프트웨어
    - 미들웨어의 역할 및 특징
      1. 통로 역할 : 분산 컴퓨팅 환경에서 서로 다른 하드웨어, 프로토콜, 통신 환경을 연결
      2. 표준화: 표준화된 인터페이스를 제공하여 시스템 간 데이터 교환에 일관성을 보장
      3. 위치 투명성(Location Transparency): 사용자가 데이터의 물리적 위치를 몰라도 서비스 이용이 가능
      4. 연결성: 시스템들을 1:1, 1:N, N:M 등 다양한 형태로 연결할 수 있음

2.  **미들웨어(Middleware)의 종류**
    - | 종류                       | 핵심 키워드               | 주요 특징 및 사례                                                                  |
      | -------------------------- | ------------------------- | ---------------------------------------------------------------------------------- |
      | DB (데이터베이스)          | 2-Tier, 벤더 제공         | 클라이언트와 원격 DB를 연결. (ODBC, JDBC, Glue)                                    |
      | RPC (원격 프로시저 호출)   | 로컬 호출처럼             | 다른 컴퓨터에 있는 프로시저를 마치 내 컴퓨터에 있는 것처럼 호출. (Entera, ONC/RPC) |
      | MOM (메시지 지향)          | 비동기, 메시지 기반       | 메시지 큐를 이용해 데이터를 주고받음. 즉시 응답보다 신뢰성 중시. (IBM MQ, JMS)     |
      | TP-Monitor (트랜잭션 처리) | 항공/철도 예약, 빠른 응답 | 대량의 트랜잭션을 감시 및 제어. 사용자 수가 늘어도 응답 속도 유지. (Tuxedo, Tmax)  |
      | ORB (객체 요청 브로커)     | 객체 지향, CORBA          | 코바(CORBA) 표준 스펙을 구현하여 객체 간의 통신을 지원. (Orbix, CORBA)             |
      | WAS (웹 애플리케이션 서버) | 동적 콘텐츠, 웹 환경      | 클라이언트의 요청에 따라 동적인 데이터를 처리. (WebLogic, WebSphere, JEUS, Tomcat) |

3.  **미들웨어 솔루션 식별 및 명세서 작성**
    - 개발 및 운영 환경에 적합한 미들웨어를 고르고 상세 내역을 적는 과정
    1. 미들웨어 솔루션 식별
       - 방법: 아키텍처 구성 정보와 소프트웨어 구매 내역을 확인하여 도입할 솔루션을 확정
       - 목록 작성: 시스템명, 구분, 솔루션명, 버전, 제조사 등을 정리
       - 검토: 이해관계자에게 전달하여 누락이나 오류가 없는지 최종 확인
    2. 미들웨어 솔루션 명세서 작성
       - 상세 기술: 식별된 미들웨어의 상세 정보, 제공 기능, 특징을 기술
       - 환경 검토: 제품 안내서나 담당자를 통해 사용 환경, 제약 사항을 꼼꼼히 확인
       - 포함 내용: 제품명, 버전, 사용 목적, 지원 범위, 기술 지원 체계 등.
