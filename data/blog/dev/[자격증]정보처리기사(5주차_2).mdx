### 1. 단위 모듈 구현

1.  **단위 모듈(Unit Module)의 개요**
    1. 단위 모듈
       - 소프트웨어 구현에 필요한 여러 동작 중 하나의 동작을 수행하는 최소 단위 모듈
       - 단위 모듈이 수행하는 하나의 기능 → 단위 기능
       - 사용자 또는 다른 모듈로부터 값을 전달받아 실행되는 작은 프로그램
    2. 특징
       - 두 개의 단위 모듈 결합 → 두 개의 기능 구현 가능
       - 독립적인 컴파일 가능
       - 다른 모듈에 호출되거나 삽입가능
       - 구성요소
         1. 처리문
         2. 명령문
         3. 데이터 구조

2.  **단위 모듈 구현 절차**
    - 단위 기능 명세서 작성 → 입·출력 기능 구현 → 알고리즘 구현
    1. 단위 기능 명세서 작성
       - 설계 단계에서 작성되는 기능 명세서, 코드 명세서, 설계 지침을 의미하며 단위 기능을 명확히 정의한 문서
       - 작성시 고려사항
         1. 추상화 작업 필요 : 복잡한 시스템을 단순화
         2. 구조화 작업 필요 : 대형 시스템을 분해해 단위 기능별 분류를 하고, 계층적 구조를 구성
         3. 정보 은닉 : 모듈 간의 독립성 유지, 한 모듈이 다른 모듈에 영향을 주지 않게 설계해야함
    2. 입·출력 기능 구현
       - 단위 기능 명세서에서 정의한 데이터 형식을 기준으로 입력과 출력 처리 로직을 구현
       - 작성시 고려사항
         1. 단위 모듈 간 데이터 전달
         2. 모듈 간 연동 및 통신
         3. 사용자 인터페이스(CLI, GUI)와의 연결
         4. 네트워크 통신이나 외부 장치와의 입·출력이 필요한 경우, Open Source API를 활용하면 보다 효율적으로 구현가능
       - IPC(Inter Process Communication)
         - IPC는 여러 개의 프로세스가 동시에 실행되는 환경에서 프로세스 간 데이터 교환을 가능하게 하는 통신 방식
         - IPC 방식의 사례
           1. Shared Memory : 여러 프로세스가 하나의 메모리를 공유하여 통신
           2. Socket : 네트워크를 통해 프로세스 간 통신
           3. Semaphore : 공유 자원에 대한 접근을 제어하여 동기화 수행
           4. Pipe / Named Pipe : FIFO 구조의 메모리를 이용한 통신 방식
           5. Message Queue : 메시지 단위로 데이터를 전달하는 방식
    3. 알고리즘 구현
       - 앞에서 정의한 입·출력 데이터를 바탕으로 단위 기능별 요구 사항을 실제 코드로 구현
       - 단순히 코드를 작성하는 것이 아니라 요구 사항을 제대로 만족하는지 확인하는 과정까지 포함
       - 단위 모듈의 종류
         1. 디바이스 드라이버 모듈 : 하드웨어 주변 장치 제어
         2. 네트워크 모듈 : 네트워크 장비 및 데이터 통신 처리
         3. 파일 모듈 : 파일 및 데이터 구조 접근 기능
         4. 메모리 모듈 : 가상 메모리 매핑 및 해제, 프로세스 간 통신
         5. 프로세스 모듈 : 프로세스 생성 및 관리 기능

### 2. 단위 모듈 테스트

1.  **단위 모듈 테스트의 개요**
    1. 단위 모듈 테스트
       - 하나의 단위 모듈이 정해진 기능을 정확하게 수행하는지 검증하는 테스트
       - 단위 모듈 테스트는 단위 테스트(Unit Test)라고도 불리며 화이트박스 테스트와 블랙박스 테스트 기법을 함께 사용한다.
    2. 단위 모듈 테스트의 필요성
       - 에러를 아주 이른 단계에서 발견할 수 있게함

2.  **테스트 케이스(Test Case)**
    - 단위 모듈 테스트에서 가장 중요한 개념
    - 소프트웨어가 요구사항을 정확히 만족하는지 확인하기 위해 설계된 테스트 항목의 명세서
    1. 테스트 케이스의 필요성
       - 테스트 케이스 없이 테스트를 진행하면 아래와 같은 문제가 발생
         1. 특정 기능에 대한 검증이 누락됨
         2. 불필요한 테스트를 반복하게 됨
         3. 테스트 결과가 사람마다 달라짐
       - 테스트 케이스 구성 요소
         1. 식별자(Identifier) : 테스트 케이스를 구분하기 위한 ID 또는 일련번호
         2. 테스트 항목(Test Item) : 테스트 대상이 되는 모듈 또는 기능
         3. 입력 명세(Input Specification) : 입력 데이터 또는 테스트 조건
         4. 출력 명세(Output Specification) : 테스트 수행 시 기대되는 결과
         5. 환경 설정(Environmental Needs) : 테스트에 필요한 하드웨어나 소프트웨어 환경
         6. 특수 절차 요구(Special Procedure Requirement) : 테스트 수행 시 특별히 필요한 절차
         7. 의존성 기술(Inter-case Dependencies) : 다른 테스트 케이스와의 의존 관계

### 3. 개발 지원 도구

1.  **통합 개발 환경(IDE: Integrated Development Environment)**
    - 프로그램 개발에 필요한 모든 작업을 하나의 프로그램에서 처리할 수 있도록 제공하는 개발 환경
    - 과거에는 편집기(Editor), 컴파일러(Compiler), 디버거(Debugger)를 각각 따로 사용했지만, 현재는 이 모든 기능을 하나의 인터페이스로 통합해서 제공하는 것이 일반적
    - 대표적인 통합 개발 환경 도구
      1. Eclipse
         - 크로스 플랫폼 지원
         - Java 기반 개발에 많이 사용
         - C, C++, PHP, Python 등 다양한 언어 지원
      2. Visual Studio
         - Microsoft에서 개발
         - Windows 환경에 최적화
         - C, C++, Python 등 지원
      3. Xcode
         - Apple에서 제공
         - macOS, iOS 개발 전용 IDE
      4. Android Studio
         - Google에서 제공
         - Android 앱 개발 전용
         - 공식 빌드 도구로 Gradle 사용
      5. IntelliJ IDEA
         - JetBrains에서 개발
         - Java, Kotlin, Go, PHP 등 다양한 언어 지원
         - 강력한 코드 분석과 자동 완성 기능 제공
    - IDE가 제공하는 주요 기능
      1. 코딩(Coding)
         - C, Java, Python 등의 프로그래밍 언어로 프로그램을 작성하는 기능
      2. 컴파일(Compile)
         - 고급 언어로 작성된 소스 코드를
         - 컴퓨터가 이해할 수 있는 목적 프로그램으로 변환
      3. 디버깅(Debugging)
         - 프로그램 실행 중 발생하는 오류(버그)를 찾아 수정하는 기능
      4. 배포(Deployment)
         - 완성된 소프트웨어를 사용자에게 전달하는 기능

2.  **빌드(Build)와 빌드 도구**
    - 빌드(Build)란 소스 코드 파일들을 실행 가능한 소프트웨어로 변환하는 과정 또는 그 결과물을 의미
    - 이 과정에는 전처리, 컴파일, 라이브러리 결합 등의 작업이 포함된다
    - 대표적인 빌드 도구
      1. Ant
         - 아파치 소프트웨어 재단에서 개발
         - XML 기반 빌드 스크립트 사용
         - 자유도와 유연성이 높음
         - 표준 규칙이 없어 재사용성이 떨어짐
      2. Maven
         - Ant의 단점을 보완하여 개발
         - 규칙과 표준이 존재
         - 의존성(Dependency) 관리 가능
         - 컴파일과 빌드를 동시에 수행
      3. Gradle
         - Ant와 Maven을 보완한 빌드 도구
         - Android Studio의 공식 빌드 도구
         - Groovy 기반 스크립트 사용
         - Maven과 동일하게 의존성 관리 가능
         - 완성된 소프트웨어를 사용자에게 전달하는 기능

3.  **협업 도구**
    - 협업 도구 여러 사람이 서로 다른 환경에서 원활하게 프로젝트를 수행하도록 도와주는 도구
    - 협업 소프트웨어 또는 그룹웨어(Groupware)라고도 부름
    - 이 과정에는 전처리, 컴파일, 라이브러리 결합 등의 작업이 포함된다
    - 협업 도구의 종류
      - 프로젝트 및 일정 관리
        - 프로젝트 진행 상황과 일정 공유
        - 예시: Google Calendar, Trello, Jira, Flow 등
      - 정보 공유 및 커뮤니케이션
        - 주제별 채널을 만들어 대화 및 파일 공유
        - 예시: Slack, Jandi, Taskworld 등
      - 디자인 협업
        - 디자이너의 UI 설계를 개발자에게 전달
        - 예시: Sketch, Zeplin
      - 기타 협업 도구
        - Evernote: 아이디어 및 메모 공유
        - Swagger: API 문서화 도구
        - GitHub: Git 기반 형상 관리 및 협업 플랫폼
