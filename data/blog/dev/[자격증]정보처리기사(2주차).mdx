### 1. 소프트웨어 공학 - 요구사항 확인

5.  **현행시스템 분석(As-Is System Analysis)**
    1.  개요
        - **현행시스템 분석(As-Is System Analysis)**이란, 현재 조직에서 사용 중인 업무 절차, 정보 흐름, 시스템 기능, 데이터 구조, 기술 환경 등을 조사·분석하여 현행 시스템의 구조와 운영 실태를 정확히 파악하는 활동이다.
        - 신규 시스템(To-Be) 구축 또는 시스템 개선을 위한 기초 자료를 확보하고, 개발 범위와 이행 방향성을 설정하기 위한 필수 단계이다.
    2.  현행시스템 분석의 목적
        - 개발 대상 시스템의 개발 범위 및 대상 명확화 - 현행 시스템을 분석
        - 현행 시스템의 구조와 문제점을 파악하여 개선 방향 도출 - 현행 시스템의 개선 도모
        - 신규 시스템 구축 시 이행 전략(To-Be) 수립의 기준 제공 - 시스템 변경 구축 방안 계획
        - 시스템 전환에 따른 위험 요소 사전 식별 - 시스템 전환 디버깅
    3.  현행시스템 파악 절차
        - 현행시스템 분석은 시스템을 구성하는 요소를 단계적으로 파악하기 위해 다음과 같은 절차에 따라 수행한다.
          1.  **[1단계]** - 시스템 구성·기능 및 인터페이스 현황 파악
              - 시스템 구성 파악
                - 현행 시스템이 어떤 하위 시스템들로 구성되어 있는지 식별
                - 각 하위 시스템의 역할과 책임 범위 파악
              - 시스템 기능 파악
                - 각 시스템이 제공하는 주요 기능 및 업무 처리 범위 파악
                - 중복 기능 및 불필요한 기능 식별
              - 시스템 인터페이스 현황 파악
                - 타 시스템과의 정보 교환 방식 및 연계 구조 파악
                - 인터페이스는 사용자 UI가 아닌, 시스템·컴포넌트·단위 업무 처리 모듈 간의 상호작용과 연관성을 의미함
              - **[시스템 요소]** 주요 인터페이스 구성 요소
                - EAI(Enterprise Application Integration, 기업 어플리케이션 통합)
                  - 기업 내의 컴퓨터 애플리캐이션들을 현대화하고, 통합하고, 조정하는 것을 목표로 세운 계획, 방법 및 도구를 의미
                - FEP (Front-End Processor, 전처리기)
                  - 메인 시스템(Host/Server) 앞단에서 통신 처리와 입·출력 제어를 전담하는 시스템
                  - 주요 목적
                    - 메인 시스템의 부하 감소
                    - 통신 효율 및 안정성 향상
                    - 보안 및 접근 제어
                    - 이기종 시스템 간 연계 지원
          2.  **[2단계]** - 시스템 아키텍처 및 소프트웨어 구성 파악
              - 시스템 아키텍처 파악
                - 시스템 내 상위 시스템과 하위 시스템 간의 구조적 관계와 상호작용을 분석
                - 시스템의 전체 구조, 행위, 행위 원리를 중심으로 시스템이 어떻게 동작하는지를 표현
                - 시스템 목적 달성을 위해 구성된 각 컴포넌트를 식별하고, 컴포넌트 간 정보 교환 방식과 흐름을 파악
                - 단위 업무 시스템별 아키텍처가 상이한 경우, 핵심 기간 업무 처리 시스템을 기준으로 분석 수행
              - 소프트웨어 구성 파악
                - 시스템 내 업무 처리용 소프트웨어에 대해 다음 사항을 명시
                  - 소프트웨어 품명, 용도, 라이선스 적용 방식, 라이선스 수량
                - 라이센스 적용 단위
                  - 사이트 단위, 서버 단위, 프로세서/ 코어 단위, 사용자 수 기반 등
          3.  **[3단계]** - 하드웨어·네트워크 및 개발 기술 환경 파악
              - 하드웨어 구성 파악
                - 서버 사양 : CPU 처리 속도 / 메모리 용량 / 저장 장치(디스크) 용량
                - 서버 이중화 구성 여부
                  - 장애 발생 시 서비스 지속 가능성 확인
                  - 현행 시스템에 이중화가 적용된 경우, 목표 시스템에서도 이중화 요구 가능성 검토
                  - 이에 따른 기술 난이도 및 비용 증가 요소 분석
              - 네트워크 구성 파악
                - 네트워크 구조 및 구성 방식
                - 시스템 간 통신 경로 및 연결 관계
              - 개발 기술 환경 분석
                - 개발 대상 시스템의 기술 환경 분석
                  - 플랫폼
                    - 응용 소프트웨어 + 하드웨어 + 소프트웨어
                    - 다양한 애플리케이션이 작동하는 기본이 되는 운영체제 소프트웨어
                    - 종류 : JAVA, .NET, IOS, Android, Windows
                    - 기능 : 개발 / 운영 / 유지보수 비용의 감소, 생산성 향상, 동일 플랫폼 간의 네트워크 효과
                    - 플렛폼 성능 특성 분석
                      - 현행 시스템의 사용자가 요구사항을 통하여 시스템 성능상의 문제를 요구할 경우, 플랫폼 성능 분석을 통하여 사용자가 느끼는 속도를 파악하고 개선하는 방향을 제시
                      - 특성 분석 항목
                        - 응답시간, 가용성, 사용률
                      - 플랫폼 성능 특성 분석 방법
                        - 기능 테스트 : 현재 시스템의 플랫폼을 평가할 수 있는 기능 테스트를 수행
                        - 사용자 인터뷰 : 사용자를 대상으로 현행 플랫폼 기능의 불편함을 파악
                        - 문서 점검 : 플랫폼과 유사한 플랫폼의 기능 자료를 분석
                  - 운영체제(OS)
                    - OS(Operation System) 정의 및 기능
                      - HW, SW 자원 관리 및 공통 서비스 제공, 사용자와의 인터페이스를 제공
                      - 종류 : Windows, Android, IOS, UNIX,LINUX, MAC 등
                    - 현행시스템의 OS 분석 항복 및 고려사항 분석
                      - 현재 정보 시스템의 OS 종류와 버전, 패치 일자, 백업주기 분석
                      - 고려사항 : 가용성, 성능, 기술 지원, 주변기기, 구축비용
                        - TCD(Total Cost of Ownership) : 일정 기간 자산 획득에 필요한 직,간접적인 총비용 HW, SW 구매비용, 운영 교육, 기술 지원, 유지보수, 손실, 에너지 사용 비용
                        - 현행 환경 분석 과정에서 라이선스의 종류, 사용자 수, 기술의 지속 가능성을 고려해야 한다.
                      - 현행 환경 분석 과정에서 라이센스의 종류, 사용자수, 기술의 지속 가능성 등을 고려해야 한다.
                      - 메모리 누수 : 실행 SW가 정상 종료되지 않고 남아있는 증상
                  - DBMS
                    - 분석 항목: DBMS 종류, 버전, 구성 방식, 저장 용량, 백업 주기, 테이블 수량, 데이터 증가 추이
                    - 대표 DBMS: Oracle, IBM DB2, Microsoft SQL Server, MySQL 등
                    - 분석시 고려사항
                      - 가용성
                      - 성능
                      - 기술 지원
                      - 상호호환성
                      - 구축비용
                  - 미들웨어(Middleware)
6.  **요구사항 개발**
    1.  요구 공학
        - 요구공학은 시스템 또는 소프트웨어 개발 과정에서 요구사항을 식별·분석·정의하고, 이를 문서화하여 지속적으로 관리하는 전반적인 프로세스를 의미한다.
        - 핵심 역할 및 효과
          - 이해관계자 간 효과적인 의사소통을 통해 공통된 이해를 형성한다.
          - 요구사항 누락 및 오해를 줄여 불필요한 재작업과 비용을 절감할 수 있다.
          - 요구사항 변경 이력을 체계적으로 관리하여 추적성과 투명성을 확보한다.
          - 분석 결과를 문서로 남겨 유지보수 및 확장 시 중요한 기준 자료로 활용할 수 있다.
        - 산출물 및 활용
          - 요구사항 명세서, 자료 흐름도, 자료 사전 등의 산출물을 통해 시스템의 구조와 동작을 명확하고 구체적으로 표현할 수 있다.
          - 필요에 따라 상태 다이어그램, 명세서 기반 정의 등 다양한 표현 기법을 활용한다.
        - 요구공학의 목적
          - 소프트웨어 개발 시 이해관계자 간 원활한 협업과 의사소통 환경 제공
          - 요구사항 누락 및 상호 이해 오류 제거를 통한 개발 리스크 최소화
          - 요구사항 변경 관리로 개발 비용 및 일정 효율화
          - 비용·일정 제약을 고려한 타당성 분석 및 요구사항 정의의 체계화
        - 요구사항의 베이스라인
          - 이해 당사자 간의 명시적 합의 내용, 프로젝트 목표 달성 여부 확인
          - 요구사항을 조기에 명확이 확정하고, 추후 발생 가능한 변경사항을 체계적으로 관리하기 위한 기준
        - 요구공학(개발) 프로세스
          - 요구사항 도출 : 이해관계자로부터 요구사항을 수집하고 정리
          - 요구사항 분석 : 요구사항의 모호성, 충돌 여부, 실현 가능성 분석
          - 요구사항 명세 : 분석된 요구사항을 명확하고 일관된 문서로 작성
          - 요구사항 검증 : 요구사항이 정확하고 완전한지 최종 확인
    2.  기능적/비기능적 요구사항
        - 기능적 요구사항 (Functional Requirements) - 기능에 대한 내용
          - 시스템이 무엇을 해야 하는지에 대한 요구사항
          - 사용자의 요청에 따라 시스템이 실제로 수행해야 할 기능을 정의
          - 입력 → 처리 → 출력 관점에서 시스템 동작을 설명
          - 예시
            - 사용자는 회원가입을 할 수 있어야 한다.
            - 관리자는 사용자 정보를 조회·수정·삭제할 수 있어야 한다.
            - 차량 대여 시스템은 대여 가능 차량 목록을 제공해야 한다.
        - 비기능적 요구사항 (Non-Functional Requirements) - 기능 이외의 성능이나, 보안에 대한 내용
          - 시스템이 어떻게 동작해야 하는지에 대한 요구사항
          - 성능, 보안, 품질, 안정성 등 기능 수행을 지원하는 조건을 정의
          - 시스템의 전반적인 **품질 속성(Quality Attributes)**에 해당
          - 주요 항목
            - 성능(Performance)
            - 보안(Security)
            - 신뢰성(Reliability)
            - 가용성(Availability)
            - 확장성(Scalability)
          - 예시
            - 차량 대여 시스템의 모든 화면은 3초 이내에 사용자에게 표시되어야 한다.
            - 시스템은 사용자 비밀번호를 암호화하여 저장해야 한다.
            - 장애 발생 시 10분 이내에 복구되어야 한다.
