### 1. 서버 개발

1.  **서버 개발이란?**

    > **한 줄 정의:** 웹 애플리케이션 로직을 구현할 **서버 프로그램을 제작**하여 **WAS에 탑재**하는 것

2.  **서버 개발 핵심 개념**

    | 항목      | 내용                              |
    | --------- | --------------------------------- |
    | 목적      | 웹 애플리케이션 **로직 구현**     |
    | 탑재 위치 | **WAS** (웹 애플리케이션 서버)    |
    | 역할      | 웹 서버 요청 처리 → **결과 반환** |
    - 서버 개발 언어
      - Java, JavaScript, Python, PHP, Ruby 등

3.  **서버 개발 프레임워크**

    > **한 줄 정의:** 네트워크 설정, 요청/응답 처리, 아키텍처 구현을 **손쉽게 처리**하도록 지원하는 소프트웨어
    - 특징
      - 대부분 **MVC 패턴** 기반으로 개발
      - 지원 언어가 **제한적** → 선정 프레임워크도 제한적
      - 서버 개발 프레임워크 종류 (시험 필수!)
        | 프레임워크 | 언어 | 특징 |
        | :---------------: | :------------: | ---------------------------------------------- |
        | **Spring** | **Java** | **전자정부 표준** 프레임워크 기반 |
        | **Node.js** | **JavaScript** | **비동기 I/O**, 이벤트 위주, **실시간** 적합 |
        | **Django** | **Python** | 컴포넌트 **재사용**, **신속한 개발** |
        | **Codeigniter** | **PHP** | 인터페이스 **간편**, 서버 자원 **적게** 사용 |
        | **Ruby on Rails** | **Ruby** | 테스트 웹 서버 지원, DB 작업 **단순화/자동화** |

4.  **서버 프로그램 구현**
    - 구현 방식

      ```
      모듈 개발 → 공통 모듈 개발 → 모듈 통합
      ```

5.  **모듈 (Module)**
    - 모듈화를 통해 분리된 시스템의 각 **기능 단위**
      | 동의어 |
      | ---------- |
      | 서브루틴 |
      | 서브시스템 |
      | 프로그램 |
      | 작업 단위 |
    - 모듈 독립성
      - 특정 모듈 수정이 **다른 모듈에 영향 미치지 않음**

      | 독립성 높이려면 |        방향         |
      | --------------- | :-----------------: |
      | **결합도**      | **낮게** (약하게) ↓ |
      | **응집도**      | **높게** (강하게) ↑ |
      | **모듈 크기**   |     **작게** ↓      |

    - 공통 모듈
      - 여러 프로그램에서 **재사용(Reuse)**할 수 있는 모듈

      | 예시                     |
      | ------------------------ |
      | 자주 사용되는 **계산식** |
      | **사용자 인증** 기능     |

6.  **웹 시스템 구조**

    ```
    [사용자] ↔ [웹 서버] ↔ [WAS] ↔ [DB]
                            ↑
                      서버 프로그램
                      (로직 구현)
    ```

### 2. 보안 및 API

1. **소프트웨어 개발 보안이란?**

   > **한 줄 정의:** 개발 과정에서 **보안 취약점을 최소화**하여 안전한 SW를 개발하기 위한 **보안 활동**

2. **소프트웨어 개발 보안 목표**

   |    목표    | 의미                 |
   | :--------: | -------------------- |
   | **기밀성** | 인가된 사용자만 접근 |
   | **무결성** | 데이터 변조 방지     |
   | **가용성** | 필요할 때 사용 가능  |

   💡 **기무가** = 기밀성 + 무결성 + 가용성 (CIA)

3. **소프트웨어 개발 보안 점검 항목 8가지**

   | 점검 항목                    | 한 줄 정의                            |
   | ---------------------------- | ------------------------------------- |
   | **세션 통제**                | 세션 **연결 및 정보 관리**            |
   | **입력 데이터 검증 및 표현** | 입력 데이터 **유효성 검증**           |
   | **보안 기능**                | 인증, 접근제어, **암호화**            |
   | **시간 및 상태**             | **병렬 처리** 환경에서 시간/상태 관리 |
   | **에러 처리**                | 오류 사전 정의, **문제 예방**         |
   | **코드 오류**                | 형 변환, **자원 반환** 고려           |
   | **캡슐화**                   | 데이터와 함수를 **하나로 묶음**       |
   | **API 오용**                 | 취약한 API **사용 방지**              |
   - 점검 항목별 상세 내용
     1. 세션 통제
        - 서버-클라이언트 **연결(세션)** 및 정보 관리

        | 보안 약점                        |
        | -------------------------------- |
        | 불충분한 세션 관리               |
        | 잘못된 세션에 의한 **정보 노출** |

     2. 입력 데이터 검증 및 표현
        - 입력 데이터 **유효성 검증** 체계 구축

        | 보안 약점                      |
        | ------------------------------ |
        | **SQL 삽입** (SQL Injection)   |
        | 경로 조작 및 자원 삽입         |
        | **크로스사이트 스크립팅(XSS)** |

     3. 보안 기능
        - 인증, 접근제어, 기밀성, **암호화** 기능

        | 보안 약점                      |
        | ------------------------------ |
        | 적절한 인증 없는 중요기능 허용 |
        | **부적절한 인가**              |

     4. 시간 및 상태
        - **병렬 처리/다중 프로세스** 환경에서 시간/상태 관리

        | 보안 약점                                   |
        | ------------------------------------------- |
        | **TOCTOU** (검사 시점과 사용 시점) 경쟁조건 |
        | 종료되지 않는 반복문/재귀함수               |

     5. 에러 처리
        - 오류 **사전 정의**하여 문제 예방

        | 보안 약점                        |
        | -------------------------------- |
        | 오류 메시지를 통한 **정보 노출** |
        | 오류 상황 **대응 부재**          |

     6. 코드 오류
        - **형 변환, 자원 반환** 등 고려하며 코딩

        | 보안 약점              |
        | ---------------------- |
        | **널 포인터 역참조**   |
        | 부적절한 **자원 해제** |

     7. 캡슐화
        - 데이터와 함수를 **하나의 객체로 묶음**

        | 보안 약점                           |
        | ----------------------------------- |
        | 잘못된 세션에 의한 데이터 정보 노출 |
        | 제거되지 않은 **디버그 코드**       |

     8. API 오용
        - **취약한 API 사용 방지**

        | 보안 약점                        |
        | -------------------------------- |
        | **DNS lookup**에 의존한 보안결정 |
        | 취약한 API 사용                  |

4. **API (Application Programming Interface)**

   > **한 줄 정의:** 라이브러리를 이용할 수 있도록 **규칙을 정의**해 놓은 인터페이스

5. **API 특징**
   - 프로그래밍 언어에서 **특정 작업 수행**
   - 운영체제 기능 활용 (파일 제어, 화상 처리, 문자 제어)
   - 개발에 필요한 **도구 제공** → 쉽고 효율적 구현

6. **API 종류**

   | API 종류                  | 설명               |
   | ------------------------- | ------------------ |
   | **Windows API**           | Windows 운영체제용 |
   | **단일 유닉스 규격(SUS)** | Unix 표준          |
   | **Java API**              | Java 언어용        |
   | **웹 API**                | 웹 서비스용        |
   | **Open API**              | **무료 공개** API  |

### 3. 배치 프로그램

1. **배치 프로그램이란?**

   > **한 줄 정의:** 사용자 상호작용 없이 **미리 정해진 순서**에 따라 **일괄 처리**하는 프로그램

2. **배치 프로그램 종류 (주기별)**

   |        종류        | 한 줄 정의                    | 예시            |
   | :----------------: | ----------------------------- | --------------- |
   |   **정기 배치**    | **정해진 기간**에 정기적 수행 | 일, 주, 월 배치 |
   | **이벤트성 배치**  | **특정 조건 충족** 시 수행    | 조건 만족 시    |
   | **On-Demand 배치** | **사용자 요청** 시 수행       | 요청 시 실행    |

3. **배치 프로그램 필수 요소 5가지**

   | 요소              | 한 줄 정의                                  |
   | ----------------- | ------------------------------------------- |
   | **대용량 데이터** | 대량 데이터 처리 가능                       |
   | **자동화**        | 사용자 개입 **없이** 수행                   |
   | **견고성**        | 잘못된 데이터로 **중단 없이** 수행          |
   | **안정성/신뢰성** | 오류 **추적** 가능 (위치, 시간)             |
   | **성능**          | 지정 시간 내 처리, 다른 프로그램 **방해 X** |

4. **배치 스케줄러 (Batch Scheduler)**

   > **한 줄 정의:** 배치 작업이 설정된 주기에 맞춰 **자동 수행**되도록 지원하는 도구
   - 다른 명칭
     - **잡 스케줄러** (Job Scheduler)
   - 대표 배치 스케줄러
     - **스프링 배치** (Spring Batch)
       - Spring 프레임워크 기반의 **오픈소스 배치 프레임워크**
       - 특징
         - 2007년 **Spring Source + Accenture** 공동 개발
         - 스프링 프레임워크 기능 **모두 사용** 가능
         - DB/파일 데이터 교환용 **컴포넌트** 제공
         - 로그 관리, 트랜잭션, 작업 재시작 등 지원
       - 스프링 배치 구성 요소 (시험 필수!)
         | 구성 요소 | 역할 |
         | :----------------: | ----------------------------------- |
         | **Job** | 수행할 **작업 정의** |
         | **Job Launcher** | **실행**을 위한 인터페이스 |
         | **Step** | Job 처리를 위한 **제어 정보** |
         | **Job Repository** | 작업 실행을 위한 **모든 정보 저장** |
         ```
         Job Repository (모든 정보 저장)
               ↓
         Job Launcher (실행)
               ↓
            Job (작업 정의)
               ↓
            Step (제어 정보)
         ```

     - **Quartz**
       - 스프링 프레임워크용 배치 처리 **오픈소스 라이브러리**
       - 특징
         - 작업과 수행 시간을 **분리** → 유연성 제공
       - Quartz 구성 요소 (시험 필수!)

         |   구성 요소   | 역할                     |
         | :-----------: | ------------------------ |
         | **Scheduler** | **실행환경** 관리        |
         |    **Job**    | 수행할 **작업 정의**     |
         | **JobDetail** | Job의 **상세 정보**      |
         |  **Trigger**  | Job 실행 **스케줄 정의** |

         ```
         Scheduler (실행환경 관리)
               ↓
         Trigger (스케줄 정의)
               ↓
         JobDetail (상세 정보)
               ↓
            Job (작업 정의)
         ```

     - 스프링 배치 vs Quartz 비교

       |   구분    |            스프링 배치             |       Quartz       |
       | :-------: | :--------------------------------: | :----------------: |
       |   유형    |           **프레임워크**           |   **라이브러리**   |
       |  개발사   |     Spring Source + Accenture      |      오픈소스      |
       | 공통 요소 |                Job                 |   Job, Scheduler   |
       | 고유 요소 | Job Launcher, Step, Job Repository | JobDetail, Trigger |
