### 1. 모듈 간 공통 기능 및 데이터 인터페이스 확인

1. **모듈 간 공통 기능 및 데이터 인터페이스의 개요**
   - 모듈 간의 연계를 설정하기 위해 가장 먼저 수행해야 할 작업
     - 공통 기능 (Common Function): 여러 모듈에서 공통적으로 제공되거나 사용하는 기능을 의미
     - 데이터 인터페이스 (Data Interface): 모듈 간 데이터를 주고받을 때 사용되는 **파라미터(매개변수)**를 의미
     - 활용: 인터페이스 설계서를 바탕으로 기능을 식별하며, 이는 향후 인터페이스 상세 설계의 기초

2. **인터페이스 설계서의 종류**
   | 구분 | 주요 내용 | 세부 구성 |
   | -------------- | ----------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
   | 일반적 설계서 | 텍스트 중심의 문서 | - 시스템 인터페이스 설계서: 전체 목록 및 데이터 명세<br/>- 상세 기능별 인터페이스 명세서: 세부 기능, 사전/사후 조건, 반환값 |
   | 정적·동적 모형 | 다이어그램(UML 등) 중심 | - 구성 요소 간의 트랜잭션을 통해 흐름 파악<br/>- 시스템의 어느 부분에 속하는지 시각적으로 확인 |

3. **모듈별 기능 확인**
   - 설계서의 내용을 보고 어느 쪽이 **외부 모듈(송신)**이고 어느 쪽이 **내부 모듈(수신)**인지 구분하는 것이 핵심
     - 시스템 인터페이스 목록: 송신/전달 쪽은 외부, 수신 쪽은 내부.
     - 상세 기능 인터페이스 명세서
       - 외부 모듈: 오퍼레이션(Operation), 사전 조건(Pre-condition)
       - 내부 모듈: 사후 조건(Post-condition)
     - 정적/동적 모형: 인터페이스 영역을 기준으로 상위/하위 모듈을 구분하여 식별.

### 2. 모듈 연계를 위한 인터페이스 기능 식별

1. **모듈 연계의 개요**
   - 모듈 연계란 내부 모듈과 외부 모듈, 또는 내부 모듈 간의 데이터 교환을 위해 관계를 설정하는 것을 의미
   - 대표적인 연계 방식으로는 EAI와 ESB가 있음
     1. EAI (Enterprise Application Integration)
        - EAI는 기업 내부의 다양한 애플리케이션 및 플랫폼 간의 정보 전달과 통합을 가능하게 하는 솔루션
        - EAI의 주요 구축 유형
          1. Point-to-Point: 1:1로 직접 연결하는 가장 기초적인 방식입니다. 변경이나 재사용이 어렵다는 단점
          2. Hub & Spoke: 중앙에 허브(Hub)를 두고 모든 데이터를 처리하는 중앙 집중형 방식. 확장과 유지보수가 쉽지만, 허브가 고장 나면 전체 시스템이 마비
          3. Message Bus (ESB 방식): 애플리케이션 사이에 미들웨어를 두어 처리합니다. 확장성이 뛰어나고 대용량 데이터 처리에 유리
          4. Hybrid: 그룹 내부는 Hub & Spoke, 그룹 간에는 Message Bus 방식을 사용하는 혼합형. 데이터 병목 현상을 최소화할 수 있음.
     1. ESB (Enterprise Service Bus)
        - ESB는 애플리케이션 간의 연계, 데이터 변환 등을 **표준 기반의 인터페이스(웹 서비스 등)**로 제공하는 솔루션
        - ESB의 특징
          1. 느슨한 결합 (Loosely Coupled): 특정 서비스에 종속되지 않도록 결합도를 낮게 유지하여 범용성을 높힘.
          2. 관리 효율성: 보안 및 관리 유지가 쉽고 높은 수준의 품질을 지원

2. **모듈 연계 기능 식별 절차**
   - 실제로 어떤 기능을 연계할지 찾아내는 과정
     1. EAI (Enterprise Application Integration)
        - 연계 기능 식별: 공통 기능과 데이터 인터페이스를 기반으로 시나리오를 작성하여 구체화
        - 인터페이스 기능 식별
          - 외부 모듈의 요청에 따라 수행되는 기능을 확인.
          - 업무 시나리오를 통해 내부 모듈이 결과값을 어떻게 내보내는지 정의.
        - 최종 선별: 식별된 기능 중 실제 구현에 필요한 기능을 최종적으로 확정.

### 3. 모듈 간 인터페이스 데이터 표준 확인

1. **인터페이스 데이터 표준의 개요**
   - 모듈 간 인터페이스에 사용되는 데이터의 형식을 표준화하는 것을 의미
   - 기존 데이터에서 공통 영역을 추출하거나, 한쪽의 데이터를 변환하여 정의
   - 확인된 표준은 향후 인터페이스 기능 구현을 정의하는 기초 자료가 됨

2. **데이터 표준 확인 절차**
   - 데이터 표준은 크게 세 단계를 거쳐 최종 확정됨
     1. 데이터 인터페이스 확인: 식별된 데이터 인터페이스를 통해 입·출력값의 의미와 특성을 확인
     2. 인터페이스 기능 확인: 인터페이스 기능 구현에 필요한 추가 데이터 항목이 있는지, 수정이나 삭제가 필요한지 검토
     3. 최종 인터페이스 데이터 표준 확인: 위 두 과정을 통해 확인된 항목들을 검토하여 최종 표준을 확정하고 출처를 기록

3. **입·출력 데이터 표준 사례**
   | 구분 | 파라미터 | 의미 (Key 값 및 내용) | 데이터 표준 (형식/규격) | 출처 |
   |------|----------|---------------------|------------------------------------------|-----------|
   | 입력값 | 급여 코드 | 급여 작업의 단위를 나타냄 | 숫자 6자리(연월), 정규직 R/계약직 T, 증가 시 숫자 증가 | 데이터 인터페이스 |
   | 입력값 | 급여 계산 결과 | 직원별 급여/총액/공제액 정보 | 사번, 소속, 직급, 급여, 상여, 세금, 실지급액 등 항목 | 데이터 인터페이스 |
   | 출력정보 | 전표 정보 | 전표 매입/매출 및 세금 신고 여부 | 발생 시기(YYYYMMDD 8자리), 전표 구분(매입 AP, 매출 AR) | 데이터 인터페이스 |
   | 출력정보 | 차·대변 검증 | 계정별 마스터/디테일 정보 발생 | 부서별, 계정별 항목 수와 총금액 검증 | 인터페이스 기능 |

4. **인터페이스 기능 구현을 위한 필요 데이터 항목**
   - 인터페이스 기능에 따라 추가로 확인해야 할 상세 데이터 항목
     - 전표 발생 수신 시 (입력값): 전표 일자, 계정, 금액, 발의 부서, 귀속 부서, 거래처, 지급 여부 등.
     - 전표 발생 결과 수신 시 (결과값): 전표 발생 여부, 차·대변 검증 여부, 부서별/계정별 금액 등.

### 4. 인터페이스 구현

1. **인터페이스 구현의 정의**
   - 인터페이스 구현은 송·수신 시스템 간의 데이터 교환 및 처리를 실제로 실현하는 작업
   - 정의된 기능을 바탕으로 구현 방법과 범위를 분석하여 최적의 방식을 선택하는 것

2. **데이터 통신을 이용한 구현**
   - 애플리케이션 영역에서 데이터 포맷을 생성하여 전송하고, 수신 측에서 이를 파싱(Parsing)하여 해석하는 방식
   - 핵심 기술: JSON, XML, AJAX
     - JSON (JavaScript Object Notation): 속성-값 쌍으로 이루어진 데이터 객체를 전달하는 텍스트 기반의 개방형 표준 포맷. 사람이 읽기 쉽고 XML보다 가볍다는 특징.
     - XML (eXtensible Markup Language): 다목적 마크업 언어로, HTML의 호환성 문제와 SGML의 복잡함을 해결하기 위해 개발.
     - AJAX (Asynchronous JavaScript and XML): 자바스크립트를 사용해 클라이언트와 서버 간에 XML 데이터를 비동기로 교환하는 기술. 페이지 전체를 새로고침 하지 않고도 인터랙티브한 상호작용이 가능.
   - JSON 구현 순서
     1. 데이터 선택: 시스템 환경에 맞게 인터페이스할 데이터를 선택.
     2. 객체 생성: 선택한 데이터를 JSON 형식의 인터페이스 객체로 변환
     3. 데이터 전송: AJAX 기술 등을 사용하여 수신 측으로 객체를 보냄.
     4. 파싱 및 처리: 수신 측에서 객체를 수신하여 파싱 후 업무를 처리.
     5. 결과 반환: 처리가 완료되면 송신 측에 처리 결과를 회신

3. **인터페이스 엔티티(테이블)를 이용한 구현**
   - 시스템 사이에 별도의 인터페이스 엔티티(테이블)를 두어 데이터를 공유하는 방식
   - 구조: 송신 테이블과 수신 테이블을 각각 두거나 하나를 공유하며, 두 테이블의 구조는 대개 동일하게 설계합.
   - 장점: 데이터 이력을 관리하기 쉽고, 대량의 데이터를 안정적으로 처리할 때 유리
   - 구현 순서
     1. 데이터 기록: 이벤트 발생 시 송신측 인터페이스 테이블에 데이터를 Write.
     2. 데이터 전송: 정해진 주기에 따라 수신측 테이블로 데이터를 보냄.
     3. 데이터 읽기: 수신측에서 입력된 데이터를 정해진 주기에 따라 Read.
     4. 트랜잭션 수행: 읽어온 데이터를 바탕으로 사전에 정의된 데이터 처리를 완료.

### 5. 인터페이스 보안

1. **인터페이스 보안 개요 및 취약점 분석**
   - 인터페이스는 모듈 간 통신 통로이므로 보안이 취약할 경우 시스템 전체에 악영향을 줄 수 있음. 따라서 인터페이스 보안성 향상을 위해 각 구간의 구현 현황을 확인하고 취약점을 분석 필요.
     - 구간별 현황 확인: 송·수신 영역의 구현 기술 및 특징을 구체적으로 확인.
     - 취약점 시나리오 작성: 송신 데이터 선택, 객체 생성, 송·수신 과정, 결과 전송 등 각 영역에서 발생할 수 있는 보안 위협을 시나리오 형태로 분석.

2. **영역별 인터페이스 보안 기능 적용**
   - 분석된 취약점을 바탕으로 네트워크, 애플리케이션, 데이터베이스의 세 가지 영역에 보안 기능을 적용
     | 적용 영역 | 주요 보안 대책 및 기술 |
     |--------|---------------------------------------------------------------------------|
     | 네트워크 | 스니핑(Sniffing) 등을 통한 데이터 탈취 방지를 위해 트래픽 암호화 설정 (IPSec, SSL, S-HTTP 등 방식 활용) |
     | 애플리케이션 | 소프트웨어 개발 보안 가이드(시큐어 코딩)를 참조하여 코드 상의 보안 취약점을 보완 및 설계 |
     | 데이터베이스 | 접근 권한(스키마, 엔티티) 관리, 프로시저 및 트리거 보안 강화, 민감 데이터 암호화 및 익명화 |

3. **핵심 보안 용어 정리**
   - 능동적/수동적 공격 유형과 개발 보안 개념
     - 스니핑(Sniffing): 네트워크 중간에서 패킷 정보를 도청하는 수동적 공격의 일종. 대부분의 패킷이 암호화되어 있지 않은 점을 이용.
     - 소프트웨어 개발 보안(시큐어 코딩): 설계 및 구현 단계에서 보안 취약점을 발견·제거하고 보안 기능을 고려하여 소프트웨어를 개발하는 일련의 활동.

4. **데이터 무결성 검사 도구**
   - 시스템 파일의 변경 유무를 확인하여 백도어 설치나 허가받지 않은 파일 수정을 감지하는 데 사용
     - 작동 원리: 해시(Hash) 함수를 이용해 현재 파일/디렉토리 상태를 DB에 저장한 후, 감시 중에 상태가 달라지면 관리자에게 알립
     - 대표적인 도구 종류:
       - Tripwire (가장 대표적)
       - AIDE, Samhain
       - Claymore, Slipwire, Fcheck

### 6. 인터페이스 구현 검증

1. **인터페이스 구현 검증 및 감시 개요**
   - 구현된 인터페이스가 설계대로 잘 작동하는지 '확인(Test)'하고 지속적으로 '감시(Monitoring)'하는 행위
   - 이를 위해 검증 도구와 **감시 도구(APM)**를 사용하며, 인터페이스 명세서의 세부 기능을 참조하여 적절한 도구를 선택
     - 구현 검증: 인터페이스가 정상적으로, 문제없이 작동하는지 확인하는 과정
     - 구현 감시: 인터페이스가 서비스를 제공하는 동안 정상 동작 여부나 에러 발생 여부를 지속적으로 체크하는 과정

2. **인터페이스 구현 검증 도구**
   - 인터페이스의 단위 기능과 시나리오를 기반으로 한 통합 테스트를 수행하기 위해 자동화된 테스트 프레임워크를 사용
     | 도구명 | 주요 특징 및 기능 |
     |----------|----------------------------------------------------------------------------|
     | xUnit | 동일한 테스트 코드를 반복 작성하지 않게 돕는 단위 테스트 프레임워크입니다. (JUnit, CppUnit, NUnit 등) |
     | STAF | 서비스 호출 및 컴포넌트 재사용 등 다양한 환경을 지원하며, 크로스 플랫폼 및 분산 소프트웨어 테스트를 지원합니다. |
     | FitNesse | 웹 기반으로 테스트케이스 설계, 실행, 결과 확인 등을 지원하는 테스트 프레임워크입니다. |
     | NTAF | **FitNesse(협업)**와 **STAF(재사용)**의 장점을 통합한 **NHN(Naver)**의 테스트 자동화 프레임워크입니다. |
     | Selenium | 다양한 브라우저와 개발 언어를 지원하는 웹 애플리케이션 전용 테스트 프레임워크입니다. |
     | watir | Ruby 언어를 사용하는 애플리케이션 테스트 프레임워크입니다. |

3. **인터페이스 구현 감시 도구 (APM)**
   - 애플리케이션 성능 관리(APM) 도구를 통해 시스템 부하, 트랜잭션 수행 내역 등을 실시간으로 분석
   - APM의 주요 기능 및 종류
     - 기능: 접속자 지원 현황, 트랜잭션 수행 내역, 장애 진단 등 종합 모니터링을 제공
     - 리소스 방식: 하드웨어 자원 중심 모니터링. (예: Nagios, Zabbix, Cacti)
     - 엔드투엔드(End-to-End) 방식: 사용자부터 DB까지의 전체 흐름 중심 모니터링

4. **검증 및 감시 확인 절차**
   - 실제로 도구를 사용해 확인해야 할 사항
   - 인터페이스 구현 검증 확인
     - 최초 입력값부터 최종 처리 결과까지의 데이터가 예상 결과값과 일치하는지 비교.
     - 각 단계별로 **오류 처리(Exception Handling)**가 적절하게 구현되어 있는지 확인.
   - 인터페이스 구현 감시 확인
     - 감시 도구가 제공하는 리포트를 활용하여 에러 발생 여부와 동작 상태를 확인.
     - 외부 시스템과의 연계 모듈이 중단 없이 정상적으로 서비스를 제공하는지 체크.
