### 1. Secure SDLC

1. **Secure SDLC란?**

   > **한 줄 정의:** **SDLC에 보안 강화를 위한 프로세스**를 포함하여 보안상 안전한 소프트웨어를 개발하는 것
   - SDLC (Software Development Life Cycle) : 소프트웨어가 만들어지고, 쓰이다가, 사라질 때까지의 전 과정

2. **Secure SDLC 개요**

   | 항목      | 내용                                    |
   | --------- | --------------------------------------- |
   | 목적      | **보안상 안전한** 소프트웨어 개발       |
   | 등장 배경 | 유지 보수 단계에서 **보안 비용 최소화** |
   | 범위      | SDLC **전체 단계**에 보안 활동 수행     |

3. **Secure SDLC 적용 단계**

   | 단계          |
   | ------------- |
   | 요구사항 분석 |
   | 설계          |
   | 구현          |
   | 테스트        |
   | 유지 보수     |

4. **Secure SDLC 대표적인 방법론**

   |        방법론         |     개발사      | 특징                                                  |
   | :-------------------: | :-------------: | ----------------------------------------------------- |
   |       **CLASP**       | Secure Software | **초기 단계** 보안 강화, **활동 중심**, **역할 기반** |
   |        **SDL**        |  **Microsoft**  | 기존 SDLC 개선, **나선형 모델** 기반                  |
   | **Seven Touchpoints** |        -        | **7개** 보안 강화 활동 수행                           |
   1. CLASP
      - SDLC **초기 단계**에서 보안을 강화하기 위해 개발된 방법론
      - CLASP 특징

        | 특징   | 설명                                  |
        | ------ | ------------------------------------- |
        | 개발사 | **Secure Software**                   |
        | 단계   | **초기 단계** 보안 강화               |
        | 구성   | **활동 중심**, **역할 기반** 프로세스 |
        | 적합   | **현재 운용 중인 시스템**에 적용      |

   2. SDL
      - **마이크로소프트**에서 개발한 안전한 소프트웨어 개발 방법론
      - SDL 특징

        | 특징   | 설명               |
        | ------ | ------------------ |
        | 개발사 | **Microsoft**      |
        | 기반   | **나선형 모델**    |
        | 목적   | 기존 SDLC **개선** |

   3. Seven Touchpoints
      - SDLC 각 단계에 **7개의 보안 강화 활동**을 수행하는 방법론
      - Seven Touchpoints 특징

        | 특징 | 설명                               |
        | ---- | ---------------------------------- |
        | 핵심 | **7개** 보안 강화 활동             |
        | 범위 | 모든 **산출물**에 위험 분석/테스트 |
        | 통합 | **모범사례**를 SDLC에 통합         |

5. **보안 요소**

   > **한 줄 정의:** 소프트웨어 개발에 있어 **충족시켜야 할 보안 요소 및 요건**
   - 보안 3대 요소 (CIA)

     |    요소    |      영문       | 한 줄 정의                           |
     | :--------: | :-------------: | ------------------------------------ |
     | **기밀성** | Confidentiality | **인가된 사용자에게만** 접근 허용    |
     | **무결성** |    Integrity    | **인가된 사용자만** 수정 가능        |
     | **가용성** |  Availability   | 인가된 사용자는 **언제든** 사용 가능 |

   - 기타 보안 요소

     |     요소      |      영문       | 한 줄 정의                   |
     | :-----------: | :-------------: | ---------------------------- |
     |   **인증**    | Authentication  | **합법적인 사용자**인지 확인 |
     | **부인 방지** | Non-Repudiation | **송·수신 사실 부인 불가**   |

6. **보안 3대 요소 상세**
   1. 기밀성 (Confidentiality)
      - **인가된 사용자에게만** 접근 허용

      | 특징    | 설명                             |
      | ------- | -------------------------------- |
      | 접근    | **인가된 사용자**만              |
      | 노출 시 | 데이터 **읽을 수 없음** (암호화) |

   2. 무결성 (Integrity)
      - **인가된 사용자만** 정보 수정 가능

      | 특징 | 설명                     |
      | ---- | ------------------------ |
      | 수정 | **인가된 사용자**만 가능 |
      | 보장 | 정보의 **정확성/완전성** |

   3. 가용성 (Availability)
      - 인가된 사용자는 **언제든지** 사용 가능

      | 특징 | 설명                   |
      | ---- | ---------------------- |
      | 사용 | **언제라도** 사용 가능 |
      | 보장 | 서비스 **연속성**      |

7. **SDLC 단계별 보안 활동**

   |       단계        | 보안 활동                                      |
   | :---------------: | ---------------------------------------------- |
   | **요구사항 분석** | 보안 **요구사항 식별**, 보안 정책 문서화       |
   |     **설계**      | 보안 설계서 작성, **위협 식별**, 보안통제 기준 |
   |     **구현**      | **시큐어 코딩**, 단위 테스트, 코드 점검        |
   |    **테스트**     | 동적 분석, **모의 침투테스트**, 결과 문서화    |
   |   **유지보수**    | 보안사고 식별, **보안 패치** 실시              |
   1. 요구사항 분석 단계 보안 활동
      - 보안 항목에 해당하는 **요구사항을 식별**하는 작업
      - 요구사항 분석 단계 활동

      | 활동                           |
      | ------------------------------ |
      | 보안 **요구사항 식별**         |
      | 보안 요소별 **등급 구분/분류** |
      | 보안 정책 **문서화**           |

   2. 설계 단계 보안 활동
      - 보안 요구사항을 설계서에 반영하고 **보안 설계서 작성**
      - 설계 단계 활동

        | 활동                               |
        | ---------------------------------- |
        | **보안 설계서** 작성               |
        | **위협 식별** 및 보안대책 수립     |
        | 환경에 대한 **보안통제 기준** 수립 |

      - 설계 단계 보안통제 기준

        |       환경        | 보안통제                                     |
        | :---------------: | -------------------------------------------- |
        |   **네트워크**    | 네트워크 **분리**, **방화벽** 설치           |
        |     **서버**      | 보안 OS, 보안 업데이트, **접근통제**         |
        |  **물리적 보안**  | **출입통제**, 개발 공간 제한, CCTV           |
        | **개발 프로그램** | 허가되지 않은 프로그램 통제, **무결성 검사** |

   3. 구현 단계 보안 활동
      - **시큐어 코딩** 가이드를 준수하며 보안 요구사항 구현
      - 구현 단계 활동

        | 활동                            |
        | ------------------------------- |
        | **표준 코딩 정의서** 준수       |
        | **시큐어 코딩**                 |
        | 지속적인 **단위 테스트**        |
        | **코드 점검** 및 소스 코드 진단 |
        - 시큐어 코딩 (Secure Coding)
          - 구현 단계에서 **보안 취약점을 최소화**하기 위해 보안 요소를 고려하며 코딩하는 것
          - 시큐어 코딩 특징

          | 특징 | 설명                          |
          | ---- | ----------------------------- |
          | 목적 | **보안 취약점 최소화**        |
          | 효과 | **안정성/신뢰성** 확보        |
          | 준비 | **시큐어 코딩 가이드** 작성   |
          | 교육 | 개발 참여자에게 **교육 실시** |

   4. 테스트 단계 보안 활동
      - 보안 설계서를 바탕으로 보안 사항이 **정확히 반영/동작되는지 점검**
      - 테스트 단계 활동

      | 활동                               |
      | ---------------------------------- |
      | **동적 분석** 도구 사용            |
      | **모의 침투테스트** 수행           |
      | 위협 **해결 여부 검증**            |
      | 모든 결과 **문서화** 및 **피드백** |

   5. 유지보수 단계 보안 활동
      - 발생 가능한 보안사고를 식별하고 **보안 패치 실시**
      - 유지보수 단계 활동

      | 활동                  |
      | --------------------- |
      | 보안사고 **식별**     |
      | 사고 발생 시 **해결** |
      | **보안 패치** 실시    |

### 2. 세션 통제

1. **세션 통제란?**

   > **한 줄 정의:** 서버와 클라이언트의 **세션 연결과 정보를 관리**하는 것
   - 세션(Session) : 클라이언트에서 서버로 접근한 이 사용자가 누구인지 기억하기 위해 유지하는 상태 정보

2. **세션 통제 개요**

   | 항목          | 내용                               |
   | ------------- | ---------------------------------- |
   | **세션**      | 서버와 클라이언트의 **연결**       |
   | **세션 통제** | 세션 연결과 정보 **관리**          |
   | 점검 단계     | **요구사항 분석** 및 **설계** 단계 |

3. 세션 통제의 보안 약점

   | 보안 약점                        |
   | -------------------------------- |
   | **불충분한 세션 관리**           |
   | **잘못된 세션에 의한 정보 노출** |

4. **불충분한 세션 관리**

   > **한 줄 정의:** 세션ID가 **일정한 규칙**으로 발급되거나 **타임아웃이 너무 길게** 설정된 경우 발생하는 보안 약점
   - 불충분한 세션 관리 원인

     | 원인                          |
     | ----------------------------- |
     | 세션ID에 **일정한 규칙** 존재 |
     | **타임아웃** 너무 길게 설정   |

   - 불충분한 세션 관리 위험

     | 위험 | 설명                               |
     | ---- | ---------------------------------- |
     | 공격 | **세션 하이재킹**                  |
     | 결과 | 인가되지 않은 시스템 **기능 이용** |
     | 결과 | 중요한 **정보 접근**               |

   - 세션 하이재킹 (Session Hijacking)
     - 서버에 접속 중인 클라이언트의 **세션 정보를 가로채는** 공격 기법 (세션 가로채기)
     - 세션 하이재킹 과정

       | 순서 | 과정                                     |
       | :--: | ---------------------------------------- |
       |  1   | 정상 연결을 **RST(Reset) 패킷**으로 종료 |
       |  2   | 재연결 시 **공격자에게 연결**            |
       |  3   | **시퀀스 번호** 이용하여 시스템 접근     |

     - 세션 하이재킹 탐지 방법 (시험 필수!)

       | 탐지 방법                        |
       | -------------------------------- |
       | **비동기화 상태** 탐지           |
       | **ACK Storm** 탐지               |
       | **패킷의 유실** 탐지             |
       | **예상치 못한 접속의 리셋** 탐지 |

5. **잘못된 세션에 의한 정보 노출**

   > **한 줄 정의:** **다중 스레드** 환경에서 **멤버 변수**에 정보를 저장할 때 발생하는 보안 약점
   - 잘못된 세션 정보 노출 특징

     | 특징 | 설명                               |
     | ---- | ---------------------------------- |
     | 환경 | **다중 스레드 (Multi-Thread)**     |
     | 원인 | **멤버 변수**에 정보 저장          |
     | 관련 | **싱글톤 패턴**의 **레이스컨디션** |
     | 결과 | **동기화 오류**, 정보 노출         |

   - 방지 방법

     | 방법                                  |
     | ------------------------------------- |
     | **멤버 변수** 대신 **지역 변수** 활용 |
     | 변수의 **범위 제한**                  |

6. **세션 설계 시 고려 사항**

   | 고려 사항          | 설명                                      |
   | ------------------ | ----------------------------------------- |
   | **로그아웃 UI**    | 모든 페이지에서 **로그아웃 가능**하도록   |
   | **세션 완전 제거** | 로그아웃 시 세션 **완전 제거**            |
   | **타임아웃 설정**  | 중요도 높음: **2~5분**, 낮음: **15~30분** |
   | **이전 세션 종료** | 이전 세션 종료 후 새 세션 생성            |
   | **중복 로그인**    | 중복 접근에 대한 **정책 수립**            |
   | **패스워드 변경**  | 활성 세션 **삭제 후 재할당**              |

7. **세션ID 관리 방법**

   | 관리 방법     | 설명                           |
   | ------------- | ------------------------------ |
   | **길이**      | 최소 **128비트**               |
   | **예측 불가** | 안전한 **난수 알고리즘** 적용  |
   | **노출 방지** | **URL Rewrite** 사용 ❌        |
   | **로그인 시** | 이전 세션ID **삭제 후 재할당** |
   | **장기 접속** | **주기적 재할당**              |

8. **세션ID 관리 핵심**

   |   핵심   | 설명              |
   | :------: | ----------------- |
   | **길이** | **128비트** 이상  |
   | **생성** | **난수** 알고리즘 |
   | **갱신** | **주기적 재할당** |

9. **세션 보안 약점 비교표**

   |           약점            | 원인                       | 방지                  |
   | :-----------------------: | -------------------------- | --------------------- |
   |  **불충분한 세션 관리**   | 규칙적 세션ID, 긴 타임아웃 | 난수, 적절한 타임아웃 |
   | **잘못된 세션 정보 노출** | 멤버 변수, 다중 스레드     | **지역 변수** 사용    |

### 3. 입력 데이터 검증 및 표현

1. **입력 데이터 검증 및 표현이란?**

   > **한 줄 정의:** 입력 데이터로 인해 발생하는 문제를 예방하기 위해 **구현 단계**에서 검증해야 하는 **보안 점검 항목**

2. **입력 데이터 검증 및 표현 개요**

   | 항목      | 내용                             |
   | --------- | -------------------------------- |
   | 목적      | 입력 데이터로 인한 **문제 예방** |
   | 점검 단계 | **구현 단계**                    |
   | 핵심      | **유효성 검증** 체계 갖추기      |

3. **입력 데이터 검증 방법**

   | 방법                        |
   | --------------------------- |
   | **유효성 검증** 체계 갖추기 |
   | **자료형** 올바른지 확인    |
   | 일관된 **언어 셋** 사용     |

4. **입력 데이터 검증 및 표현의 보안 약점**

   |          보안 약점          | 한 줄 정의                            |
   | :-------------------------: | ------------------------------------- |
   |        **SQL 삽입**         | SQL 삽입하여 **DB 유출/변조**         |
   | **경로 조작 및 자원 삽입**  | 경로 조작하여 **서버 자원 수정/삭제** |
   |           **XSS**           | 악의적인 **스크립트 삽입**            |
   |  **운영체제 명령어 삽입**   | 시스템 명령어로 **권한 탈취**         |
   | **위험한 형식 파일 업로드** | 악성 **스크립트 파일 업로드**         |
   |    **신뢰되지 않는 URL**    | **피싱 사이트**로 유도                |
   |  **메모리 버퍼 오버플로**   | **버퍼 범위 초과** 읽기/쓰기          |

5. **SQL 삽입 (SQL Injection)**

   > **한 줄 정의:** 웹 응용 프로그램에 **SQL을 삽입**하여 DB 데이터를 **유출/변조**하고 **관리자 인증을 우회**하는 보안 약점
   - SQL 삽입 특징

     | 특징 | 설명                            |
     | ---- | ------------------------------- |
     | 대상 | **동적 쿼리**                   |
     | 목적 | DB **유출/변조**, 인증 **우회** |

   - SQL 삽입 방지

     | 방지 방법                       |
     | ------------------------------- |
     | 입력 데이터에 **예약어** 필터링 |
     | **특수문자** 필터링             |

6. **경로 조작 및 자원 삽입**

   > **한 줄 정의:** 데이터 **입출력 경로를 조작**하여 서버 자원을 **수정/삭제**할 수 있는 보안 약점
   - 경로 조작 방지

     | 방지 방법                         |
     | --------------------------------- |
     | **경로 순회 공격** 방지 필터 사용 |

7. **크로스사이트 스크립팅 (XSS)**

   > **한 줄 정의:** 웹페이지에 **악의적인 스크립트를 삽입**하여 방문자 **정보 탈취** 또는 **비정상적 기능 수행**을 유발하는 보안 약점
   - XSS 특징

     | 특징 | 설명                             |
     | ---- | -------------------------------- |
     | 영문 | **C**ross **S**ite **S**cripting |
     | 대상 | **웹페이지**                     |
     | 방법 | **악의적인 스크립트** 삽입       |
     | 목적 | 정보 **탈취**, 비정상 기능       |

   - XSS 방지

     | 방지 방법                           |
     | ----------------------------------- |
     | **HTML 태그** 사용 제한             |
     | **특수문자 치환**: `<`, `>`, `&` 등 |

   - XSS 치환 문자
     - `<`, `>`, `&` → **다른 문자로 치환**

8. **운영체제 명령어 삽입**

   > **한 줄 정의:** 외부 입력값으로 **시스템 명령어 실행**을 유도하여 **권한 탈취** 또는 **시스템 장애**를 유발하는 보안 약점
   - 운영체제 명령어 삽입 방지

     | 방지 방법                                        |
     | ------------------------------------------------ |
     | 웹 인터페이스로 **시스템 명령어 전달 차단**      |
     | 외부 입력값을 **검증 없이 내부 명령어로 사용 X** |

9. **위험한 형식 파일 업로드**

   > **한 줄 정의:** **악성 스크립트 파일을 업로드**하여 시스템 손상 또는 **시스템 제어**할 수 있는 보안 약점
   - 위험한 파일 업로드 방지 (시험 필수!)

     | 방지 방법                          |
     | ---------------------------------- |
     | 파일 **확장자 제한**               |
     | 파일명 **암호화**                  |
     | 웹사이트와 파일 서버 **경로 분리** |
     | **실행 속성 제거**                 |

10. **신뢰되지 않는 URL 주소로 자동접속 연결**

    > **한 줄 정의:** 사이트 주소를 조작하여 방문자를 **피싱 사이트로 유도**하는 보안 약점
    - URL 자동접속 방지

      | 방지 방법                                   |
      | ------------------------------------------- |
      | 외부 사이트 주소를 **화이트 리스트**로 관리 |

11. **메모리 버퍼 오버플로 (Buffer Overflow)**

    > **한 줄 정의:** 할당된 메모리의 **범위를 넘어선 위치**에서 자료를 읽거나 쓰려고 할 때 발생하는 보안 약점
    - 버퍼 오버플로 특징

      | 특징 | 설명                            |
      | ---- | ------------------------------- |
      | 발생 | **버퍼 범위 초과** 읽기/쓰기    |
      | 결과 | 프로그램 **오동작**             |
      | 위험 | 악의적 코드 실행, **권한 획득** |

    - 버퍼 오버플로 방지

      | 방지 방법                                 |
      | ----------------------------------------- |
      | **적절한 버퍼 크기** 설정                 |
      | 설정된 범위 내에서 **올바르게 읽기/쓰기** |

### 5. 예외처리

1. **에러처리란?**

   > **한 줄 정의:** 소프트웨어 실행 중 발생할 수 있는 **오류(Error)를 사전에 정의**하여 문제를 예방하기 위한 **보안 점검 항목**

2. **에러처리 개요**

   | 항목    | 내용                                        |
   | ------- | ------------------------------------------- |
   | 목적    | 오류로 인한 **문제 예방**                   |
   | 방법    | **예외처리 구문**으로 오류 정의             |
   | 미비 시 | **중요정보 노출**, 소프트웨어 **실행 중단** |
   - 에러처리 미비로 인한 문제

     | 문제                      |
     | ------------------------- |
     | **중요정보 노출**         |
     | 소프트웨어 **실행 중단**  |
     | **예기치 못한 문제** 발생 |

3. **에러처리의 보안 약점**

   |            보안 약점            | 한 줄 정의                            |
   | :-----------------------------: | ------------------------------------- |
   | **오류 메시지를 통한 정보노출** | 오류 메시지로 **중요 정보 외부 노출** |
   |     **오류 상황 대응 부재**     | **예외처리 미비**로 인한 문제         |
   |      **부적절한 예외처리**      | 오류를 **광범위하게 묶어** 처리       |

4. **오류 메시지를 통한 정보노출**
   - 오류 발생 시 **실행 환경, 사용자 정보, 디버깅 정보** 등이 외부에 노출되는 보안 약점
   - 노출 정보 종류

     | 노출 정보         |
     | ----------------- |
     | **실행 환경**     |
     | **사용자 정보**   |
     | **디버깅 정보**   |
     | **경로 정보**     |
     | **스택 트레이스** |

   - 오류 메시지 정보노출 위험

     | 위험                              |
     | --------------------------------- |
     | 해커의 **악의적인 행위** 도움     |
     | 소프트웨어 **내부구조 파악** 용이 |

   - 방지 방법

     | 방지 방법                        |
     | -------------------------------- |
     | 오류 발생 시 **내부에서만** 처리 |
     | **최소한의 정보**만 출력         |
     | **사전에 준비된 메시지**만 출력  |

5. **오류 상황 대응 부재**
   - **예외처리를 하지 않았거나 미비**로 인해 발생하는 보안 약점
   - 오류 상황 대응 부재 결과

     | 결과                        |
     | --------------------------- |
     | 소프트웨어 **실행 중단**    |
     | **의도를 벗어난 동작** 유도 |

   - 방지 방법

     | 방지 방법                                    |
     | -------------------------------------------- |
     | 오류 발생 가능 부분에 **예외처리 구문** 작성 |
     | **제어문** 활용하여 오류 악용 방지           |

6. **부적절한 예외처리**
   - 오류를 **세분화하지 않고 광범위하게** 묶어 처리하거나 **누락된 예외**가 있을 때 발생하는 보안 약점
   - 부적절한 예외처리 원인

     | 원인                                    |
     | --------------------------------------- |
     | 오류를 **광범위하게 묶어** 한 번에 처리 |
     | **누락된 예외** 존재                    |
     | 함수 반환값이 **예상과 다름**           |

   - 부적절한 예외처리 결과

     | 결과                               |
     | ---------------------------------- |
     | **예기치 않은 문제** 발생          |
     | **잘못된 값**으로 인한 다양한 문제 |

   - 방지 방법

     | 방지 방법                                  |
     | ------------------------------------------ |
     | 함수 반환값이 **의도대로 출력**되는지 확인 |
     | **세분화된 예외처리** 수행                 |

7. **예외처리 방식 비교**

   |  구분  |         잘못된 방식         |      올바른 방식       |
   | :----: | :-------------------------: | :--------------------: |
   |  범위  | **광범위** (모든 오류 묶음) | **세분화** (개별 처리) |
   | 메시지 |     **상세 정보** 출력      |  **최소한 정보** 출력  |
   |  누락  |     예외 **누락** 존재      |   **모든 예외** 처리   |

8. **예외처리 구문 예시**
   - 잘못된 예 (광범위)

     ```java
     try {
         // 코드
     } catch (Exception e) {  // 모든 예외를 한 번에 처리
         e.printStackTrace();  // 스택 트레이스 출력 (정보노출)
     }
     ```

   - 올바른 예 (세분화)

     ```java
     try {
         // 코드
     } catch (FileNotFoundException e) {
         // 파일 관련 예외 처리
     } catch (SQLException e) {
         // DB 관련 예외 처리
     } catch (Exception e) {
         // 준비된 메시지만 출력
         System.out.println("오류가 발생했습니다.");
     }
     ```

### 6. 코드 오류

1. **코드 오류란?**

   > **한 줄 정의:** **구현 단계**에서 개발자들이 실수하기 쉬운 **형(Type) 변환, 자원 반환** 등의 오류를 예방하기 위한 **보안 점검 항목**

2. **코드 오류 개요**

   | 항목      | 내용                          |
   | --------- | ----------------------------- |
   | 점검 단계 | **구현 단계**                 |
   | 대상 오류 | **형 변환**, **자원 반환** 등 |
   - 코드 오류로 인한 보안 약점

     | 보안 약점                     |
     | ----------------------------- |
     | **널 포인터 역참조**          |
     | **부적절한 자원 해제**        |
     | **해제된 자원 사용**          |
     | **초기화되지 않은 변수 사용** |

3. **코드 오류 보안 약점 요약**

   |        보안 약점         | 한 줄 정의                                |
   | :----------------------: | ----------------------------------------- |
   |   **널 포인터 역참조**   | **널 포인터**가 가리키는 메모리에 값 저장 |
   |  **부적절한 자원 해제**  | 자원 반환 코드 **누락**                   |
   |   **해제된 자원 사용**   | **반환된 메모리** 참조                    |
   | **초기화되지 않은 변수** | **초기화 안 된** 변수 사용                |

4. **널 포인터 역참조 (Null Pointer Dereference)**
   - **널 포인터가 가리키는 메모리**에 어떠한 값을 저장할 때 발생하는 보안 약점
   - 널 포인터 역참조 원인

     | 원인                                     |
     | ---------------------------------------- |
     | 라이브러리 함수가 오류 시 **널 값 반환** |
     | 반환값을 **포인터로 참조**               |

   - 널 포인터 역참조 결과

     | 결과                                |
     | ----------------------------------- |
     | 널 포인터 = **메모리 첫 주소** 참조 |
     | 소프트웨어 **비정상 종료**          |
     | 공격자가 **예외 상황 악용**         |

   - 방지 방법

     | 방지 방법                     |
     | ----------------------------- |
     | 포인터 이용 전 **널 값 검사** |

5. **택 가드 (Stack Guard)**
   - **스택**에서 발생하는 보안 약점을 막기 위해 **특정 값**을 저장하고 **변경 감지** 시 실행을 중단하는 기술
   - 스택 가드 동작 원리

     | 순서 | 동작                                     |
     | :--: | ---------------------------------------- |
     |  1   | 복귀 주소와 변수 사이에 **특정 값 저장** |
     |  2   | 해당 값이 **변경되었는지 확인**          |
     |  3   | 변경 시 **오버플로우**로 판단            |
     |  4   | 프로그램 **실행 중단**                   |

   - 스택 가드 목적

     | 목적                           |
     | ------------------------------ |
     | **잘못된 복귀 주소** 호출 방지 |
     | **버퍼 오버플로우** 방지       |

6. **부적절한 자원 해제**
   - 자원 반환 코드를 **누락**하거나 프로그램 오류로 자원을 **반환하지 못했을 때** 발생하는 보안 약점
   - 부적절한 자원 해제 원인

     | 원인                          |
     | ----------------------------- |
     | 자원 반환 코드 **누락**       |
     | 프로그램 **오류**로 반환 실패 |

   - 영향받는 자원

     | 자원                        |
     | --------------------------- |
     | **힙 메모리 (Heap Memory)** |
     | **소켓 (Socket)**           |
     | 기타 **유한한 시스템 자원** |

   - 부적절한 자원 해제 결과

     | 결과                      |
     | ------------------------- |
     | **자원 부족**             |
     | **새로운 입력 처리 불가** |

   - 방지 방법

     | 방지 방법                              |
     | -------------------------------------- |
     | 자원 반환 코드 **누락 확인**           |
     | 예외처리에 **관계없이 자원 반환** 코딩 |

7. **해제된 자원 사용**
   - 이미 **반환된 메모리를 참조**하는 경우 발생하는 보안 약점
   - 해제된 자원 사용 결과

     | 결과                         |
     | ---------------------------- |
     | **예상치 못한 값/코드** 수행 |
     | **의도하지 않은 결과** 발생  |

   - 방지 방법

     | 방지 방법                                        |
     | ------------------------------------------------ |
     | 반환된 메모리에 접근 못 하도록 **포인터 초기화** |

8. **초기화되지 않은 변수 사용**
   - 변수 선언 후 **값이 부여되지 않은 변수**를 사용할 때 발생하는 보안 약점
   - 초기화되지 않은 변수 사용 위험

     | 위험                                |
     | ----------------------------------- |
     | 이전에 사용하던 **내용이 남아있음** |
     | 변수 노출 시 **중요정보 악용** 가능 |

   - 방지 방법

     | 방지 방법                             |
     | ------------------------------------- |
     | 변수 선언 시 **할당된 메모리 초기화** |

### 7. 캡슐화

1. **캡슐화란?**

   > **한 줄 정의:** 중요한 데이터와 기능을 **불충분하게 캡슐화**하거나 **잘못 사용**함으로써 발생하는 문제를 예방하기 위한 **보안 점검 항목**
   - 중요한 데이터는 함부로 못 만지게 숨기고, 정해진 방법으로만 쓰게 하는 것

2. **캡슐화 개요**

   | 항목 | 내용                                  |
   | ---- | ------------------------------------- |
   | 목적 | **정보 은닉** 필요한 데이터/기능 보호 |
   | 문제 | **불충분한 캡슐화**, **잘못된 사용**  |

3. **캡슐화로 인한 보안 약점**

   | 보안 약점                                   |
   | ------------------------------------------- |
   | **잘못된 세션에 의한 정보 노출**            |
   | **제거되지 않고 남은 디버그 코드**          |
   | **시스템 데이터 정보 노출**                 |
   | **Public 메소드로부터 반환된 Private 배열** |
   | **Private 배열에 Public 데이터 할당**       |

4. **캡슐화 보안 약점 요약**

   |               보안 약점               | 한 줄 정의                              |
   | :-----------------------------------: | --------------------------------------- |
   |       **잘못된 세션 정보 노출**       | **다중 스레드**에서 **멤버 변수** 사용  |
   |            **디버그 코드**            | 개발 중 남겨둔 **디버그 코드**          |
   |        **시스템 데이터 노출**         | 시스템 내부 정보 **외부 출력**          |
   | **Public 메소드 → Private 배열 반환** | Private 배열을 **Public 메소드**가 반환 |
   |   **Private 배열 ← Public 데이터**    | Private 배열에 **Public 데이터** 저장   |

5. **잘못된 세션에 의한 정보 노출**

   > **한 줄 정의:** **다중 스레드** 환경에서 **멤버 변수**에 정보를 저장할 때 발생하는 보안 약점
   - 잘못된 세션 정보 노출 원인

     | 원인                                |
     | ----------------------------------- |
     | **다중 스레드 (Multi-Thread)** 환경 |
     | **멤버 변수**에 정보 저장           |
     | **싱글톤 패턴**의 **레이스컨디션**  |

   - 잘못된 세션 정보 노출 결과

     | 결과                    |
     | ----------------------- |
     | **동기화 오류** 발생    |
     | **멤버 변수 정보 노출** |

   - 방지 방법

     | 방지 방법                             |
     | ------------------------------------- |
     | **멤버 변수** 대신 **지역 변수** 활용 |
     | **변수 범위 제한**                    |

6. **제거되지 않고 남은 디버그 코드**

   > **한 줄 정의:** 개발 중 버그 수정/결과 확인을 위해 **남겨둔 디버그 코드**로 인해 발생하는 보안 약점
   - 디버그 코드 위험

     | 위험                         |
     | ---------------------------- |
     | **중요한 정보 노출**         |
     | 인증/식별 **우회 코드** 악용 |

   - 방지 방법

     | 방지 방법             |
     | --------------------- |
     | 배포 전 **코드 검사** |
     | **디버그 코드 삭제**  |

7. **시스템 데이터 정보 노출**

   > **한 줄 정의:** 시스템 내부 정보를 **시스템 메시지로 외부 출력**하도록 코딩했을 때 발생하는 보안 약점
   - 방지 방법

     | 방지 방법                                |
     | ---------------------------------------- |
     | 시스템 메시지에 **최소한의 정보**만 제공 |

   - 접근 지정자 (접근 제어자)
     - 외부로부터의 **접근을 제한**하기 위해 사용되는 **예약어**
   - 접근 지정자 범위

     |    한정자     | 클래스 내부 | 패키지 내부 | 하위 클래스 | 패키지 외부 |
     | :-----------: | :---------: | :---------: | :---------: | :---------: |
     |  **Public**   |     ⭕      |     ⭕      |     ⭕      |     ⭕      |
     | **Protected** |     ⭕      |     ⭕      |     ⭕      |     ❌      |
     |  **Default**  |     ⭕      |     ⭕      |     ❌      |     ❌      |
     |  **Private**  |     ⭕      |     ❌      |     ❌      |     ❌      |

   - 접근 지정자 범위 암기
     - **Public** = **모든 곳**에서 접근 가능
     - **Protected** = **패키지 외부 제외** (하위 클래스 ⭕)
     - **Default** = **패키지 내부**까지만
     - **Private** = **클래스 내부**만
   - 접근 범위 순서
     - **Public > Protected > Default > Private**

8. **Public 메소드로부터 반환된 Private 배열**

   > **한 줄 정의:** **Private 배열**을 **Public 메소드**에서 반환할 때 발생하는 보안 약점
   - 문제점

     | 문제                                |
     | ----------------------------------- |
     | Private 배열의 **주소가 외부 공개** |
     | **외부에서 접근 가능**              |

   - 방지 방법

     | 방지 방법                      |
     | ------------------------------ |
     | **별도의 메소드**를 통해 조작  |
     | **복제본**으로 반환 후 값 전달 |

9. **Private 배열에 Public 데이터 할당**

   > **한 줄 정의:** **Private 배열**에 **Public 데이터** 또는 메소드 파라미터를 저장할 때 발생하는 보안 약점
   - 문제점

     | 문제                                  |
     | ------------------------------------- |
     | Private 배열을 **외부에서 접근 가능** |

   - 방지 방법

     | 방지 방법                            |
     | ------------------------------------ |
     | **레퍼런스** 아닌 **값을 직접 저장** |

10. **캡슐화 보안 약점 비교표**

    |         보안 약점         | 원인           | 방지 방법          |
    | :-----------------------: | -------------- | ------------------ |
    | **잘못된 세션 정보 노출** | 멤버 변수 사용 | **지역 변수** 활용 |
    |      **디버그 코드**      | 코드 남김      | 배포 전 **삭제**   |
    |  **시스템 데이터 노출**   | 외부 출력      | **최소한 정보**    |
    | **Public → Private 배열** | 주소 반환      | **복제본** 반환    |
    |   **Private ← Public**    | 레퍼런스 저장  | **값 직접** 저장   |

11. **Private 배열 관련 약점 비교**

    | 구분 | Public 메소드 → Private 배열 | Private 배열 ← Public 데이터 |
    | :--: | :--------------------------: | :--------------------------: |
    | 방향 |    Private 배열 **반환**     |    Public 데이터 **저장**    |
    | 문제 |      배열 **주소 공개**      |    외부에서 **접근 가능**    |
    | 방지 |       **복제본** 반환        |       **값 직접** 저장       |

### 8. 암호 알고리즘 (정보처리기사)

1. **암호 알고리즘이란?**

   > **한 줄 정의:** 중요정보를 보호하기 위해 **평문을 암호화된 문장으로 만드는** 절차 또는 방법

2. **암호 알고리즘 분류**

   ```
   암호 알고리즘
   ├── 단방향 암호화 (복호화 불가)
   │   └── 해시 (Hash)
   └── 양방향 암호화 (복호화 가능)
      ├── 개인키 암호화 (대칭키)
      └── 공개키 암호화 (비대칭키)
   ```

   |    구분    |     방식      | 특징            |
   | :--------: | :-----------: | --------------- |
   | **단방향** |     해시      | **복호화 불가** |
   | **양방향** | 개인키/공개키 | **복호화 가능** |

3. **개인키 암호화 (Private Key)**
   - **동일한 키**로 데이터를 암호화하고 복호화하는 방식
   - 개인키 암호화 특징

     | 특징      | 설명                                                 |
     | --------- | ---------------------------------------------------- |
     | 다른 이름 | **대칭 암호**, **비밀키 암호**                       |
     | 키 개수   | **n(n-1)/2** (n개 대상 시)                           |
     | 장점      | **속도 빠름**, 알고리즘 **단순**, 파일 크기 **작음** |
     | 단점      | 키 수가 **많아짐** (사용자 증가 시)                  |

   - 개인키 암호화 종류

     |       분류        | 방식                             | 알고리즘                       |
     | :---------------: | -------------------------------- | ------------------------------ |
     |  **블록 암호화**  | 하나의 **데이터 블록** 암호화    | **DES, SEED, AES, ARIA, IDEA** |
     | **스트림 암호화** | **비트/바이트/워드** 단위 암호화 | **LFSR, RC4**                  |

   - 개인키 암호화 암기
     - **개인키** = **대칭키** = **비밀키** = **동일한 키**
     - 키 개수 = **n(n-1)/2**

4. **공개키 암호화 (Public Key)**
   - **공개키로 암호화**하고 **비밀키로 복호화**하는 방식
   - 공개키 암호화 특징

     | 특징      | 설명                                               |
     | --------- | -------------------------------------------------- |
     | 다른 이름 | **비대칭 암호**                                    |
     | 키 개수   | **2n** (n개 대상 시)                               |
     | 암호화 키 | **공개키 (Public Key)** - 공개                     |
     | 복호화 키 | **비밀키 (Secret Key)** - 비밀                     |
     | 장점      | 키 분배 **용이**, 키 개수 **적음**                 |
     | 단점      | **속도 느림**, 알고리즘 **복잡**, 파일 크기 **큼** |
     | 용도      | **인증**, **전자서명**                             |

   - 공개키 암호화 대표 알고리즘

     | 알고리즘  |
     | --------- |
     | **RSA**   |
     | **ECC**   |
     | **Rabin** |

   - 공개키 암호화 암기
     - **공개키** = **비대칭키** = **다른 키** (암호화/복호화)
     - 키 개수 = **2n**

5. **공개키 기반 구조 (PKI)**
   - 공개키 암호 시스템을 **안전하게 사용하고 관리**하기 위한 정보 보호 표준 방식
   - PKI 구분

     |    방식     | 설명                                    |
     | :---------: | --------------------------------------- |
     |  **X.509**  | 인증기관 발생 **인증서** 기반 상호 인증 |
     | **비X.509** | 국가/지역별 보완 및 개발                |

6. **양방향 알고리즘 종류**
   - 개인키 (대칭키) 알고리즘

     | 알고리즘 | 연도 | 개발                   | 블록/키 길이            |
     | :------: | :--: | ---------------------- | ----------------------- |
     | **SEED** | 1999 | **한국 KISA**          | 128비트 / 128, 256      |
     | **ARIA** | 2004 | **한국** 국정원+산학연 | 128비트 / 128, 192, 256 |
     | **DES**  | 1975 | **미국 NBS**           | 64비트 / 56             |
     | **AES**  | 2001 | **미국 NIST**          | 128비트 / 128, 192, 256 |

   - 한국 개발 암호 알고리즘
     - **SEED** = 1999년, **KISA** (한국인터넷진흥원)
     - **ARIA** = 2004년, **국정원+산학연** (Academy, Research Institute, Agency)
   - 공개키 (비대칭키) 알고리즘

     | 알고리즘  | 연도 | 개발                              | 특징                   |
     | :-------: | :--: | --------------------------------- | ---------------------- |
     |  **RSA**  | 1978 | **MIT** (Rivest, Shamir, Adleman) | **소인수분해** 기반    |
     |  **ECC**  | 1985 | -                                 | **타원곡선**, RSA 대안 |
     | **Rabin** | 1979 | 미하엘 라빈                       | **소인수분해** 기반    |

   - RSA 암기
     - **R**ivest, **S**hamir, **A**dleman (MIT)
     - **소인수분해** 어려움 기반

7. **해시 (Hash)**

   > **한 줄 정의:** 임의의 길이의 입력을 **고정된 길이의 값**으로 변환하는 것
   - 해시 특징

     | 특징 | 설명                         |
     | ---- | ---------------------------- |
     | 방향 | **단방향** (복호화 **불가**) |
     | 출력 | **고정된 길이**              |
     | 용도 | **무결성 검증**, 정보보호    |
     | 함수 | **일방향 함수**              |

   - 해시 함수 종류

     |   해시 함수    | 연도 | 개발              | 특징                       |
     | :------------: | :--: | ----------------- | -------------------------- |
     | **SHA 시리즈** | 1993 | **미국 NSA/NIST** | SHA-0/1/2(224,256,384,512) |
     |    **MD5**     | 1991 | R.Rivest          | 512비트/128비트            |
     |   **N-NASH**   | 1989 | **일본 NTT**      | 128비트/128비트            |
     |   **SNEFRU**   | 1990 | RC.Merkle         | 512비트/128,256            |

8. **솔트 (Salt)**
   - 암호화 수행 전 원문에 덧붙이는 **무작위 값**
   - 솔트 사용 이유

     | 이유                                        |
     | ------------------------------------------- |
     | 같은 패스워드라도 **서로 다른 결과** 생성   |
     | 하나의 암호 해제 시 **여러 계정 탈취 방지** |
     | 더 **안전한 암호화** 관리                   |
