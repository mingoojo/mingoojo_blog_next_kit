### 1. 사용자 인터페이스

1.  **사용자 인터페이스(User Interface, UI) 개요**
    - 사용자와 시스템간의 상호작용이 원할하게 이루어지도록 도와주는 장치 및 소프트웨어
    - 사용자와 백엔드간의 상호작용이 원할하게 이루어지도록 도와주는 프론트엔드

2.  **사용자 인터페이스의 특징**
    - 사용자 만족도에 가장 큰 영향을 미침 > 소프트웨어 영역중 가장 많은 변경이 발생한다.
    - 사용자의 편리성과 가독성을 높히는 걸 목적으로함.
    - 최소한의 노력으로 결과 도출할 수 있게함.
    - 사용자 중심의 설계
    - 수행 결과의 오류 제거
    - 작업기능의 구체적인 방법 제시
    - 정보 제공자와 사용자간의 매개 역할 수행
    - 사용자 인터페이스를 설계하기 위해 시스템 아키텍처의 숙지가 필요함

3.  **사용자 인터페이스의 구분**
    1. CLI(Command Line Interface) : 명령과 출력이 텍스트의 형태로 이루어짐
    2. GUI(Graphical User Interface) : 아이콘이나 메뉴를 마우스로 선택하여 작업 수행
    3. NUI(Nutural User Interface) : 사용자의 말이나 행동으로 기기조작(멀티터치 및 동작 인식)
    4. VUI(Voice User Interface) : 사람의 음성으로 기기를 조작
    5. OUI(Organic User Interface) : 모든 사물과 사용자 간의 상호작용을 위한 인터페이스.(VR/AR 인터페이스 등)

4.  **사용자 인터페이스의 기본 원칙**
    1. 직관성 : 누구나 쉽게 이해 및 사용가능
    2. 유효성 : 사용자의 목적을 달성 가능
    3. 학습성 : 쉽게 익힐 수 있어야함
    4. 유연성 : 사용자의 요구사항을 수용

5.  **사용자 인터페이스의 설계원칙**
    1. 사용자 중심 : 사용자가 쉽게 이해 및 사용할 수 있는 환경 제공
    2. 사용성 : 사용자가 쉽고 빠르게 이해할 수 있고, 효율적으로 사용할 수 있어야함
    3. 일관성 : 조작법이 일관성 있어야함
    4. 단순성 : 조작법이 단순해야 함
    5. 결과 예측 가능 : 기능의 예측과 결과가 일치해야함
    6. 심미성 : 디자인적으로 완성도가 높아야함
    7. 표준화 : 기능 구조와 디자인을 표준화하여 한번 학습으로 쉽게 사용가능해야함
    8. 접근성 : 모든 사용자가 사용할 수 있어야함
    9. 명확성 : 사용자가 개념적으로 쉽게 인지할 수 있도록 설계
    10. 오류 발생 해결 : 오류발생을 사용자가 쉽게 인지, 해결할수 있어야함

6.  **사용자 인터페이스 개발 시스템의 기능(기본적으로 제공해야하는 기능)**
    - 사용자 입력을 검증할 수 있어야함
    - 에러 처리와 에러 메세지를 표시할 수 있어야 함
    - 도움과 프롬프트를 제공해야함

### 2. UI 설계 도구

1.  **UI 설계 도구**
    - UI 구성 및 설계에 필요한 도구.
    - 와이어프레임, 목업, 스토리보드, 프로토타입, 유스케이스 등
    - 사용자의 요구사항의 구현을 기획단계에서 확인하기 위한 용도.
    1. 와이어프레임(Wireframe)
       - 페이지에 대한 대략적인 레이아웃 및 UI요소의 뼈대를 설계
       - 설계방법 : 각 페이지의 영역, 콘텐츠, 텍스트 배치등을 화면단위로 설계
       - 사용목적 : 개발자나 디자이너이 레이아웃을 협의하거나 현황 공유
       - 사용툴 : 손그림, 파워포인트, 키노트, 스케치, 일러스트, 포토샵 등
    2. 목업(Mock up)
       - 디자인, 사용방법을 보여주는 와이어프레임보다 실제화면에 가까운 형태
       - 시각적으로 구성 요소를 배치, 실제 동작하지 않음
       - 사용툴 : 파워목업, 발사믹 목업 등
    3. 스토리보드(Story Board)
       - 와이어프레임에 콘텐츠에 대한 설명 및 페이지 간의 이동의 흐름을 추가
       - 디자이너 및 개발자가 참고하는 작업 지침서.
       - 정책, 프로세스, 콘텐츠구성, 와이어프레임, 기능 정의 등의 정보를 포함
       - 사용툴 : 파워포인트, 키노트, 스케치, Axure 등
           <ImageWithCaption src="/static/images/storyboard.png" alt="story board" caption="스토리보드" />
    4. 프로토타입(Proto type)
       - 와이어프레임이나 스토리보드에 인터렉션을 추가. 실제 구현된 것처럼 테스트가 가능한 모형의 형태
       - 사용성 테스트 및 작업자 간의 서비스 이해를 위해 사용
       - 페이퍼 프로토타입, 디지털 프로토 타입으로 나뉨
       - 사용툴 : HTML/CSS, Axure, Flinto, 네이버 프로토나우, 카카오 오븐
    5. 유스케이스(Use Case)
       - 사용자 측면에서의 요구사항, 사용자가 원하는 목표달성을 위해 수행할 냐용
       - 사용자 요구사항을 빠르게 파악.
       - 사용자의 자연어로 작성된 요구사항을 구조적으로 표현 및 명세화

### 3. 품질 요구사항

1.  **품질 요구사항**
    > 소프트웨어 품질은 소프트웨어의 기능, 성능, 만족도 등 소프트웨어 특성의 총제
    - 소프트웨어가 “얼마나 잘 만들어졌는가”를 판단하기 위한 비기능적 요구사항
    1. 소프트웨어 평가 지침
       1. ISO/IEC 9126
          - 소프트웨어 품질 평가에 대한 표준지침
          - ISO/IEC 9126은 2011년에 호환성 및 보안성을 강화하여 ISO/IEC 25010으로 개정
              <ImageWithCaption src="/static/images/isoiec9126.jpg" alt="isoiec9126" caption="ISO/IEC 9126" />
       2. ISO/IEC 25010
          - 소프트웨어 품질 평가의 국제표준
              <ImageWithCaption src="/static/images/isoiec25010.jpg" alt="isoiec25010" caption="ISO/IEC 25010" />
    2. 소프트웨어 평가 특성
       1. **기능성**
          - 소프트웨어가 사용자의 요구사항에 맞게 기능하는지의 여부
          - | 상세품질 요구사항 | 설명                                                                 |
            | :---------------: | -------------------------------------------------------------------- |
            |   적합성/적절성   | 지정된 작업과 사용자의 목적달성을 위해 적절한 기능을 제공하는지 여부 |
            |   정밀성/정확성   | 사용자가 요구하는 결과를 정확하게 산출하는 능력                      |
            |    상호 운용성    | 다른 시스템과 상호작용하며 운용될 능력                               |
            |      보안성       | 정보 접근에 대한 보안성                                              |
            |      준수성       | 기능과 관련된 표준, 관례 및 규정을 준수할 능력                       |
       2. **신뢰성**
          - 소프트웨어가 일관되게 수행할 수 있는 능력
          - | 상세품질 요구사항 | 설명                                                               |
            | :---------------: | ------------------------------------------------------------------ |
            |      성숙성       | 결함으로 인한 고장을 피해갈 능력                                   |
            |    고장 허용성    | 결함 또는 인테페이스 오류 시에도 규정된 성능을 유지할 수 있는 능력 |
            |      회복성       | 고장시 규정된 성능 수준까지 회복하고, 데이터를 복구하는 능력       |
       3. **사용성**
          - 사용자가 쉽게 배우고 사용가능, 다시 사용하고 싶은 정도
          - | 상세품질 요구사항 | 설명                                                             |
            | :---------------: | ---------------------------------------------------------------- |
            |      이해성       | 소프트웨어의 적합성, 사용 방법 등을 사용자가 이해할 수 있는 능력 |
            |      학습성       | 소프트웨어를 사용자가 학습할 수 있는 능력                        |
            |      운용성       | 사용자가 소프트웨어를 제어하고 운용할 능력                       |
            |      친밀성       | 사용자가 소프트웨어를 다시 사용하고 싶도록 하는 능력             |
       4. **효울성**
          - 사용자가 요구사항을 얼마나 빠르게 처리할 수 있는지에 대한 정도
          - | 상세품질 요구사항 | 설명                                                     |
            | :---------------: | -------------------------------------------------------- |
            |    시간효율성     | 특정 기능을 수행할 때 사용되는 시간에 대한 효율성        |
            |    자원효율성     | 특정 기능을 수행할 때 사용되는 자원과 종류에 대한 효율성 |
       5. **유지보수성**
          - 새로운 환경 또는 새로운 요구사항에 대해 확장할 수 있는 능력
          - | 상세품질 요구사항 | 설명                                                 |
            | :---------------: | ---------------------------------------------------- |
            |      분석성       | 결함이나 고장의 원인을 파악하는 능력                 |
            |      변경성       | 결함 제거 및 환경변화로 인한 수정을 쉽게 구현할 능력 |
            |      안정성       | 변경으로 인한 결과의 오류를 최소화 할 수 있는능력    |
            |      시험성       | 소프트웨어의 변경을 검증할 수 있는 능력              |
       6. **이식성**
          - 새로운 환경에 정응하는지에 대한 능력
          - | 상세품질 요구사항 | 설명                                                           |
            | :---------------: | -------------------------------------------------------------- |
            |      적용성       | 다른환경으로 변화가능한지에 대한 능력                          |
            |      설치성       | 임의외 환경에 설치할 수 있는 능력                              |
            |      대체성       | 동일한 환경에서 다른 소프트웨어를 대체할 수 있는능력           |
            |      공존성       | 자원을 공유하는 환경에서 다른 소프트웨어와 공존할 수 있는 능력 |

### 4. UI 상세 설계

1.  **UI 시나리오 문서 개요**
    - UI 설계서를 바탕으로 실제 설계 및 구현을 위한 자세한 설계 단계 (반드시 시나리오 작성이 필요하다.)
    - UI 시나리오 문서
      - 인터페이스의 기능 구조, 대표화면, 인터렉션 등 다양한 상황과 예외를 문서로 정리한것
      - 각 시나리오별 최종 목표의 달성 단계 기술

2.  **UI 시나리오 문서 작성 원칙**
    1. 구체적인 작성 필요 > 계층 구조나 플로우차트 표기법으로 작성
    2. 공통기능 및 공통 UI, 공통 인터렉션을 일반 규칙으로 정의
    3. 대표화면의 레이아웃, 기능을 정의하여야 함
    4. 인터랙션의 순서, 분기, 조건, 루프 등을 정의
    5. 예외 상황에 대한 기술
    6. 기능별 상세 시나리오 정의
       <ImageWithCaption src="/static/images/uiScenario.png" alt="uiScenario" caption="ui 시나리오" />

3.  **UI 시나리오 문서 작성을 위한 일반 규칙**
    - 앱이 공통적으로 어떻게 생기고 동작하는지를 기술. 디자이너, 기획자, 개발자가 다르게 해석하지 않게하기 위함
      1.  주요 키의 위치와 기능 : 모든 화면에 공통적으로 배치되는 주요 키의 위치와 기능을 설명한 것으로 여러 화면 간의 일관성을 보장한다.
      2.  공통 UI 요소 : 체크박스, 라디오 버튼, 텍스트 박스 등의 요소를 언제, 어떤 형태로 사용할지를 정의하고 사용자가 조작하면 어떻게 반응하는지 그 흐름을 설명한다.
      3.  기본 스크린 레이아웃 : 모든 화면에 공동적으로 나타나는 Titles. Ok/Back, Soft Key, Option, Functional Buttons 등의 위치와 속성을 정의한다.
      4.  기본 인터랙션 규칙 : 터치 제스처 등에 공통적으로 사용되는 조작 방법과 실행, 이전 다음 삭제 이동 등의 화면 전환 효과 등을 기술한다.
      5.  공통 단위 태스크 흐름 : 많은 기능들에 공통적으로 사용되는 삭제 검색, 매너 모드 상태 등에 대한 인터랙션 흐름을 설명한다.
      6.  케이스 문서 : 다양한 상황에서 공통적으로 적용되는 시스템의 동작을 정의한 문서이다.

4.  **UI 시나리오 문서의 요건**
    | 요건 | 설명|
    |-----------------------------------------------|------------------------------------------------------|
    | 완전성(Complete) | - 누락되지 않도록 최대한 상세하게 기술해야 한다. <br/> - 해당 시스템의 기능보다는 사용자의 태스크에 초점을 맞춰 기술한다.|
    | 일관성(Consistent) | - 서비스 목표 시스템 및 사용자의 요구사항, UI 스타일 등이 모두 일관성을 유지해야 한다. |
    | 이해성(Understandable) | - 누구나 쉽게 이해할 수 있도록 설명한다.<br/> - 불분명하거나 추상적인 표현은 피한다. |
    | 가독성(Readlable) | - 표준화된 템플릿 등을 활용하여 문서를 쉽게 읽을 수 있도록 해야 한다.<br/> - v1.0.v2.0 등과 같이 문서 인덱스에 대한 규칙이나 목차를 제공한다.<br/> - 읽기 쉽도록 줄간격, 단락, 들여쓰기 등의 기준을 마련한다. <br/>- 시각적인 효과를 위해 여백이나 빈 페이지, 하이라이팅을 일관성 있게 지정한다. <br/> - 하이퍼링크 등을 지정하여 문서들이 서로 참조될 수 있도록 지정한다. |
    | 수정 용이성(Modifiable) | - 시나리오의 수정이나 개선이 쉬워야 한다. |
    | 추적 용이성(Traceable) | - 변경 사항은 언제, 어떤 부분이 왜 발생했는지 쉽게 추적할 수 있어야 한다. |

5.  **UI 시나리오 문서의 기대효과**
    - 요구사항이나 의사소통에 대한 오류가 감소
    - 개발 과정에서의 재작업, 혼선 최소화
    - 불필요한 기능을 최소화
    - 소프트웨어 개발 비용을 절감
    - 개발 속도 향상

### 5. HCI / UX / 감성공학

1.  **HCI(Human compoter interaction, interface)**

    > 사람과 컴퓨터가 어떻게 상호작용해야 “덜 불편하고, 덜 헷갈리고, 더 잘 쓰이냐”를 연구하는 분야
    - 최종목표는 시스템 사용에서 최적의 사용자 경험(UX)을 만드는 것

2.  **UX(User Experience)**

    > 사용자가 서비스를 이용할때 느끼는 총체적 경험
    - UX의 특징
      - 주관성 : UX의 평가는 주관적인 경향이 있음
      - 정황성 : UX의 평가는 주변환경이나 상황에 영향을 많이 받음
      - 총체성 : 개인이 느끼는 총체적인 심리적 감성적인 결과

3.  **감성공학**
    > 사람의 ‘감정·느낌·이미지’를 제품이나 서비스 설계에 반영하는 공학적 접근
    - 조금 더 쉽게 말하면 '이 버튼을 보면 안심된다 / 고급스럽다 / 재미있다' 와 같은 감정이 들게 하는 공학
    - '감성'을 과학적으로 측정하기 위해서는 생체계측 기술, 감각계측 기술, 센서, 인공지능. 생체제어 기술 등이 요구된다.
    - 감성공학의 요소기술
      - 기반기술: 제품 설계에 적용할 인간의 특성을 파악
      - 구현 기술: 인간의 특성에 맞는 인터페이스를 구현
      - 응용 기술 : 인간에 맞는지 파악하여 새로운 감성을 만든다.
