## 개요

`npm create vite@latest`로 프로젝트를 생성하면 설정 파일이 우수수 쏟아진다.

```
├── vite.config.ts
├── tsconfig.json
├── tsconfig.app.json
├── tsconfig.node.json
├── eslint.config.js
└── .prettierrc
```

각 파일이 왜 있는지, 무엇을 건드려야 하는지 정리한다.

---

## vite.config.ts

**Vite 번들러 자체의 동작**을 제어하는 파일이다.
이 파일은 브라우저가 아닌 **Node 환경**에서 실행된다.

---

### plugins

Vite에 기능을 추가하는 플러그인을 등록한다.
`@vitejs/plugin-react-swc`는 Babel 대신 Rust 기반의 SWC로 JSX를 트랜스파일하고, Fast Refresh(HMR)를 활성화한다.
SWC는 Babel 대비 컴파일 속도가 최대 20배 빠르다.

```ts
plugins: [
  react(), // SWC 기반 React 트랜스파일 + Fast Refresh
],
```

---

### resolve.alias

경로 별칭을 설정한다.
`'../../../components/Button'` 같은 상대 경로 대신 `'@/components/Button'`으로 쓸 수 있게 된다.

단, 이 설정은 **Vite(번들러)에게만** 알려주는 것이다.
TypeScript는 별도로 알려줘야 하기 때문에 `tsconfig.app.json`에도 `paths` 설정이 필요하다.

```ts
resolve: {
  alias: {
    // '@/components/Button' → 'src/components/Button'으로 치환
    // Vite(번들러)에게만 알려주는 것 — TS는 tsconfig.app.json에 별도 설정 필요
    '@': path.resolve(__dirname, './src'),
  },
},
```

### server

개발 서버의 포트, 자동 오픈, 프록시 등을 설정한다.
`proxy`는 개발 단계에서 CORS를 우회할 때 사용한다.
`/api/users`로 요청을 보내면 Vite 개발 서버가 중간에서 `http://localhost:8080/users`로 대신 전달해준다.
브라우저 입장에서는 같은 origin에 요청하는 것처럼 보이기 때문에 CORS 에러가 발생하지 않는다.

```ts
server: {
  port: 3000,   // 기본값 5173
  open: true,   // 실행 시 브라우저 자동 오픈
  proxy: {
    // '/api/users' 요청 → 'http://localhost:8080/users'로 전달
    // 개발 단계에서 CORS 우회할 때 사용
    '/api': {
      target: 'http://localhost:8080',
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api/, ''),
    },
  },
},
```

### build

프로덕션 빌드 결과물을 제어한다.
`manualChunks`로 번들을 여러 청크로 쪼갤 수 있다.
`react`, `react-dom`처럼 버전이 자주 바뀌지 않는 라이브러리를 별도 청크로 분리하면, 앱 코드가 바뀌어도 해당 청크는 브라우저가 캐시한 걸 그대로 쓰기 때문에 로딩 성능이 올라간다.

```ts
build: {
  outDir: 'dist',     // 빌드 결과물 경로 (기본값)
  sourcemap: true,    // 프로덕션 소스맵 생성 (에러 추적용)
  minify: 'esbuild',  // 'esbuild' | 'terser' | false
  rollupOptions: {
    output: {
      // 번들을 여러 청크로 분리
      // 자주 바뀌지 않는 라이브러리를 별도 청크로 두면 브라우저 캐싱 효율 상승
      manualChunks: {
        react: ['react', 'react-dom'],
        vendor: ['lodash', 'axios'],
      },
    },
  },
},
```

### define

빌드 타임에 전역 상수를 코드에 직접 박아 넣는다.
`import.meta.env`와 다르게 런타임에 값을 읽는 게 아니라, 번들러가 빌드 시점에 해당 식별자를 값으로 치환해버린다.
예를 들어 `__APP_VERSION__`은 빌드 후 번들 파일에서 `"1.0.0"` 문자열로 대체된다.
참조되지 않는 코드 분기는 번들에서 제거되기 때문에 트리쉐이킹에 유리하다.

TypeScript가 해당 식별자를 모르기 때문에 `vite-env.d.ts`에 타입 선언을 추가해야 한다.

```ts
define: {
  // 빌드 시점에 값으로 직접 치환됨
  // ex) console.log(__APP_VERSION__) → console.log("1.0.0")
  __APP_VERSION__: JSON.stringify('1.0.0'),
},
```

```ts
// vite-env.d.ts
/// <reference types="vite/client" />

declare const __APP_VERSION__: string
```

### css

CSS Module과 전처리기(Sass 등) 옵션을 설정한다.
`additionalData`에 등록한 코드는 모든 `.scss` 파일 상단에 자동으로 삽입된다.
변수나 믹스인을 매 파일마다 `@import`하지 않아도 되기 때문에 전역 스타일 토큰을 관리할 때 유용하다.

```ts
css: {
  modules: {
    // CSS Module 클래스명을 camelCase로 자동 변환
    // ex) .my-button → styles.myButton
    localsConvention: 'camelCase',
  },
  preprocessorOptions: {
    scss: {
      // 모든 .scss 파일 상단에 자동 삽입
      // → 변수, 믹스인을 매 파일마다 @import하지 않아도 됨
      additionalData: `@use "@/styles/variables" as *;`,
    },
  },
},
```

### optimizeDeps

개발 서버 첫 실행 시 Vite는 `node_modules`의 CJS 패키지를 ESM으로 변환해서 캐싱한다.
이 과정을 **사전 번들링(Pre-Bundling)** 이라고 한다.
대부분은 자동으로 처리되지만, 특정 패키지가 개발 서버에서 느리거나 에러가 반복될 때 수동으로 포함/제외 대상을 지정한다.

```ts
optimizeDeps: {
  // 자동 감지되지 않는 패키지를 명시적으로 사전 번들링 대상에 추가
  include: ['lodash-es'],
  // 사전 번들링에서 제외
  exclude: [],
},
```

## tsconfig.json / tsconfig.app.json / tsconfig.node.json

최신 Vite 템플릿은 tsconfig를 **3개로 분리**한다.

| 파일                 | 대상                | 역할                                      |
| -------------------- | ------------------- | ----------------------------------------- |
| `tsconfig.json`      | 루트                | 나머지 두 파일을 references로 묶는 진입점 |
| `tsconfig.app.json`  | `src/**`            | 브라우저에서 실행되는 React 코드          |
| `tsconfig.node.json` | `vite.config.ts` 등 | Node 환경에서 실행되는 설정 파일          |

### tsconfig.json

```json
{
  "files": [],
  "references": [
    { "path": "./tsconfig.app.json" },
    { "path": "./tsconfig.node.json" }
  ]
}
```

직접 건드릴 일은 거의 없다.

### tsconfig.app.json

실제 React 코드에 적용되는 TS 설정이다.
절대 경로 별칭은 여기에 추가한다.

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    },
    "noUnusedLocals": false,
    "noUnusedParameters": false
  }
}
```

`noUnusedLocals`를 `false`로 두는 이유는, 미사용 변수 감지는 ESLint가 담당하기 때문이다.
TS와 ESLint가 같은 걸 중복으로 잡으면 설정 충돌이 생긴다.

### tsconfig.node.json

`vite.config.ts`를 위한 별도 환경이다.
`lib`, `module`, `moduleResolution` 등이 브라우저 환경과 다르게 설정되어 있다.

## eslint.config.js

ESLint v9부터 도입된 **Flat Config** 방식이다.
기존 `.eslintrc.js` 파일은 더 이상 사용하지 않는다.

핵심은 **타입 인지(Type-Aware) 린팅**이다.
ESLint가 TypeScript 컴파일러의 타입 정보를 빌려와서 코드를 분석한다.
단순 텍스트 파싱이 아니라 타입까지 보기 때문에, 런타임에 터질 버그를 코딩 단계에서 잡아낼 수 있다.

```js
import js from '@eslint/js'
import globals from 'globals'
import reactHooks from 'eslint-plugin-react-hooks'
import reactRefresh from 'eslint-plugin-react-refresh'
import tseslint from 'typescript-eslint'
import { defineConfig, globalIgnores } from 'eslint/config'
import eslintConfigPrettier from 'eslint-config-prettier'

export default defineConfig([
  globalIgnores(['dist']),
  {
    files: ['**/*.{ts,tsx}'],
    extends: [
      js.configs.recommended,
      ...tseslint.configs.recommendedTypeChecked, // Type-Aware
      reactHooks.configs.flat.recommended,
      reactRefresh.configs.vite,
    ],
    languageOptions: {
      globals: globals.browser,
      parserOptions: {
        project: ['./tsconfig.app.json', './tsconfig.node.json'],
        tsconfigRootDir: import.meta.dirname,
      },
    },
    rules: {
      'no-console': 'warn',
      '@typescript-eslint/no-unused-vars': 'warn',
    },
  },
  eslintConfigPrettier, // 반드시 마지막에
])
```

`eslintConfigPrettier`는 Prettier와 겹치는 ESLint 포매팅 규칙을 전부 꺼준다.
배열 **맨 마지막**에 놓아야 뒤에서 덮어쓴다.

```bash
npm install -D eslint-config-prettier
```

## .prettierrc

코드 **스타일(포매팅)** 만 담당한다.
타입 검사나 코드 품질과는 무관하다.

```json
{
  "semi": false,
  "singleQuote": true,
  "tabWidth": 2,
  "trailingComma": "all",
  "printWidth": 100
}
```

> `eslint-plugin-prettier`는 설치하지 않는다.
> 포매팅 오류를 ESLint 에러로 올리는 방식인데, 저장할 때 Prettier가 자동으로 고쳐주기 때문에 불필요하다.

## VS Code 자동화 (.vscode/settings.json)

저장 시 각 도구가 자동으로 실행되도록 연결한다.

```json
{
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.formatOnSave": true,
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  },
  "typescript.tsdk": "node_modules/typescript/lib",
  "typescript.enablePromptUseWorkspaceTsdk": true
}
```

## 역할 정리

| 파일                 | 담당                                    |
| -------------------- | --------------------------------------- |
| `vite.config.ts`     | 번들러 동작 (경로 별칭, 플러그인, 빌드) |
| `tsconfig.app.json`  | 브라우저 코드 타입 검사                 |
| `tsconfig.node.json` | Node 환경(vite.config.ts) 타입 검사     |
| `eslint.config.js`   | 코드 품질, 잠재적 버그 감지             |
| `.prettierrc`        | 코드 스타일 통일                        |

세 도구가 서로 다른 레이어를 담당하기 때문에, 각자의 역할을 명확히 분리해서 설정해야 충돌이 없다.
