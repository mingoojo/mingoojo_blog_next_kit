## process.env가 뭔가

[Node.js 공식문서 — process.env](https://nodejs.org/api/process.html#processenv)

`process.env`는 **Node.js의 전역 객체**다.
현재 프로세스가 실행되는 환경의 환경 변수들을 key-value 형태로 담고 있다.

```ts
console.log(process.env.HOME)      // "/Users/jomingu"
console.log(process.env.USER)      // "jomingu"
console.log(process.env.SHELL)     // "/bin/zsh"
console.log(process.env.NODE_ENV)  // "development"
```

별도 import 없이 어디서든 접근 가능하다. Node.js가 프로세스를 시작할 때
**OS의 환경 변수를 전부 복사해서** `process.env` 객체에 넣어주기 때문이다.

여기서 중요한 건 두 가지다.

```
1. 모든 값은 "문자열"이다
   process.env.PORT = 3000  →  실제로는 "3000" (string)
   비교할 때 === 3000 하면 false다. 문자열로 비교해야 한다.

2. 브라우저에는 process가 없다
   process는 Node.js 전용 객체다.
   브라우저에서 process.env를 찍으면 "process is not defined" 에러가 난다.
```

두 번째가 핵심이다. 이게 프레임워크마다 환경 변수를 다루는 방식이 다른 근본적인 이유다.

## process.env에는 뭐가 들어있나

Next.js 프로젝트에서 `console.log(process.env)`를 찍어보면 수십 개의 변수가 나온다.
이걸 출처별로 분류하면 크게 3가지다.

### 1. OS/쉘에서 오는 것

프로세스가 시작될 때 운영체제에서 상속받는 변수들이다.
Node.js가 하는 일이 아니라 OS가 원래 가지고 있는 것들이다.

```
HOME="/Users/jomingu"            ← 홈 디렉토리
USER="jomingu"                   ← 현재 사용자
SHELL="/bin/zsh"                 ← 사용 중인 쉘
PATH="/usr/local/bin:..."        ← 실행 파일 검색 경로
LANG="C.UTF-8"                   ← 시스템 언어 설정
TERM="xterm-256color"            ← 터미널 종류
```

이 변수들은 Node.js를 쓰든 Python을 쓰든 동일하게 존재한다.

### 2. npm이 주입하는 것

`npm run dev` 같은 npm 스크립트로 실행하면
npm이 자동으로 여러 변수를 `process.env`에 추가한다.

```
npm_package_name="MJ-blog"                ← package.json의 name
npm_package_version="2.4.0"               ← package.json의 version
npm_lifecycle_event="start"               ← 현재 실행 중인 스크립트 이름
npm_lifecycle_script="next dev"           ← 실제 실행 명령어
npm_config_user_agent="npm/11.5.1..."     ← npm 버전 정보
npm_execpath="/.../npm-cli.js"            ← npm 실행 경로
```

`npm_package_*` 변수들은 `package.json`의 필드를 자동으로 읽어서 넣어주는 거라
코드에서 패키지 버전을 참조할 때 유용하다.

```ts
// 패키지 버전을 코드에서 참조
const version = process.env.npm_package_version  // "2.4.0"
```

### 3. 프레임워크가 주입하는 것

Next.js, CRA, Express 등 각 프레임워크가 자체적으로 넣어주는 변수들이다.
**이게 프레임워크마다 다른 부분이다.**

Next.js의 경우:

```
NODE_ENV="development"                    ← next dev → development, next build → production
PORT="3000"                               ← 개발 서버 포트
NEXT_RUNTIME="nodejs"                     ← 현재 런타임 (nodejs | edge)
__NEXT_PRIVATE_ORIGIN="http://localhost:3000"

# .env 파일에서 자동 로드된 값
NEXT_PUBLIC_API_URL="aaa"
NEXT_PUBLIC_SITE_NAME="MyApp"
```

## 핵심 — 브라우저에는 process가 없다

이걸 이해하면 프레임워크별 차이가 전부 설명된다.

| 구분                  | 서버 (Node.js)           | 브라우저                         |
| --------------------- | ------------------------ | -------------------------------- |
| process.env 존재 여부 | 존재함 ✅                | process 자체가 없음 ❌           |
| OS 환경 변수 접근     | 가능                     | "process is not defined" 에러    |
| 접근 가능한 변수 범위 | 모든 환경 변수 접근 가능 | 번들러가 치환해준 것만 접근 가능 |
| 값 결정 시점          | 런타임에 값이 결정됨     | 빌드 타임에 값이 고정됨          |

그러면 React 같은 프론트엔드 코드에서 `process.env.REACT_APP_API_URL`은
어떻게 동작하는 걸까?

```js
// 내가 쓴 코드
const url = process.env.REACT_APP_API_URL

// 빌드 후 번들에 들어가는 코드
const url = "https://api.example.com"
```

`process.env.REACT_APP_API_URL`이라는 코드가 **그대로 브라우저에 전달되는 게 아니라**,
빌드할 때 문자열 `"https://api.example.com"`으로 바꿔치기 된다.
브라우저는 process.env를 본 적이 없다. 이미 치환된 값만 받는 거다.

이게 **빌드 후에 환경 변수를 바꿔도 반영이 안 되는 이유**다.
치환된 문자열은 이미 번들 파일 안에 박혀있기 때문이다.
값을 바꾸려면 다시 빌드해야 한다.

## Express — process.env를 그대로 쓴다

Express는 **순수 Node.js 서버**다.
클라이언트(브라우저)에 코드를 보내는 게 아니라 서버에서 직접 실행하기 때문에,
`process.env`를 아무 제약 없이 그대로 쓸 수 있다.

```ts
// server.js
import express from 'express'

const app = express()
const port = process.env.PORT || 3000

app.get('/api/users', (req, res) => {
  // DB 비밀번호 같은 시크릿도 서버에서는 자유롭게 접근 가능
  const dbUrl = process.env.DATABASE_URL
  // ...
})

app.listen(port, () => {
  console.log(`Server running on port ${port}`)
})
```

문제는 Express는 `.env` 파일을 **자동으로 안 읽어준다**는 거다.
`.env`에 변수를 써놔도 `process.env`에 자동으로 들어가지 않는다.

### .env 파일을 읽으려면

**방법 1 — dotenv 패키지 사용**

```bash
npm install dotenv
```

```ts
// server.js 최상단에 추가
import dotenv from 'dotenv'
dotenv.config()  // .env 파일을 읽어서 process.env에 주입

console.log(process.env.DATABASE_URL)  // .env에 적은 값이 나옴
```

**방법 2 — Node.js 20+ 내장 기능 (--env-file 플래그)**

```bash
# dotenv 설치 없이 Node.js 자체 기능으로 .env 로딩
node --env-file=.env server.js

# 여러 파일도 가능 (나중 파일이 우선)
node --env-file=.env --env-file=.env.local server.js
```

**방법 3 — 커맨드라인에서 직접 주입**

```bash
PORT=4000 DATABASE_URL=postgresql://localhost/mydb node server.js
```

### Express 정리

```bash
1. .env 자동 로딩            #❌ (dotenv 또는 --env-file 필요)
2. NODE_ENV 자동 설정         #❌ (직접 설정해야 함)
3. 클라이언트 노출 접두사     #해당 없음 (서버 전용)
4. 환경 변수 접근 방식        #process.env.* 그대로
```

## CRA (Create React App) — 빌드 타임 치환

CRA는 **React 앱을 빌드해서 브라우저에 보내는** 구조다.
브라우저에는 `process`가 없으니까, CRA의 내부 Webpack 설정이
빌드할 때 `process.env.REACT_APP_*`을 문자열로 치환해준다.

```bash
# .env
REACT_APP_API_URL=https://api.example.com
SECRET_KEY=abc123
```

```tsx
// App.tsx
function App() {
  // ✅ REACT_APP_ 접두사 → 빌드 시 문자열로 치환됨
  console.log(process.env.REACT_APP_API_URL)  // "https://api.example.com"

  // ❌ 접두사 없음 → 치환 안 됨 → undefined
  console.log(process.env.SECRET_KEY)  // undefined

  // ✅ NODE_ENV는 예외적으로 항상 접근 가능
  console.log(process.env.NODE_ENV)  // "development" 또는 "production"

  return <div>Hello</div>
}
```

### REACT_APP 접두사가 필요한 이유

보안 때문이다. `.env` 파일에 DB 비밀번호나 API 시크릿이 들어있을 수 있는데,
이걸 전부 브라우저 번들에 넣어버리면 누구나 볼 수 있다.
그래서 **`REACT_APP_` 접두사가 붙은 것만** 의도적으로 클라이언트에 노출한다.

### 빌드 후 실제로 일어나는 일

```js
// 빌드 전 (내가 쓴 코드)
const api = process.env.REACT_APP_API_URL

// 빌드 후 (브라우저에 전달되는 코드)
const api = "https://api.example.com"
```

`process.env`라는 코드는 번들에 존재하지 않는다.
Webpack의 `DefinePlugin`이 빌드할 때 전부 문자열로 바꿔치기한 것이다.

### NODE_ENV 자동 설정

```bash
react-scripts start  → NODE_ENV = "development"
react-scripts build  → NODE_ENV = "production"
react-scripts test   → NODE_ENV = "test"
```

직접 바꿀 수 없다. CRA가 강제로 설정한다.

### CRA 정리

```bash
1. .env 자동 로딩            #✅ (react-scripts가 dotenv 내장)
2. NODE_ENV 자동 설정         #✅ (강제, 변경 불가)
3. 클라이언트 노출 접두사     #REACT_APP_
4. 환경 변수 접근 방식        #process.env.* (빌드 타임 치환)
```

## Vite — process.env가 아니라 import.meta.env

Vite가 다른 도구들과 **근본적으로 다른 점**이 여기 있다.

Vite는 개발 모드에서 **브라우저 네이티브 ES Module**을 그대로 사용한다.
브라우저에는 `process`가 없으니까, Vite는 아예
`process.env` 대신 **`import.meta.env`** 라는 다른 객체를 사용한다.

```bash
# .env
VITE_API_URL=https://api.example.com
DB_PASSWORD=secret
```

```tsx
// App.tsx
function App() {
  // ✅ Vite 방식 — import.meta.env 사용
  console.log(import.meta.env.VITE_API_URL)  // "https://api.example.com"

  // ❌ VITE_ 접두사 없음 → 노출 안 됨
  console.log(import.meta.env.DB_PASSWORD)  // undefined

  // ❌ process.env는 Vite에서 기본적으로 안 됨
  console.log(process.env.VITE_API_URL)  // ❌ process is not defined

  // ✅ Vite 내장 변수
  console.log(import.meta.env.MODE)  // "development" | "production"
  console.log(import.meta.env.DEV)   // true | false
  console.log(import.meta.env.PROD)  // true | false

  return <div>Hello</div>
}
```

### 왜 process.env가 아닌가

`import.meta`는 **ES Module 표준 스펙**에 있는 객체라
브라우저에서도 사용할 수 있다. `process`는 Node.js 전용이니까 브라우저에서 못 쓰지만,
`import.meta`는 브라우저가 원래 알고 있는 거다.

Vite는 이 차이를 이용해서 `import.meta.env`로 환경 변수를 노출한다.

```
CRA, Webpack
├── process.env.* 라고 쓰면
├── 빌드할 때 문자열로 "바꿔치기" 해서
└── 브라우저에서 process를 안 쓰게 됨 (trick)

Vite
├── import.meta.env.* 라고 쓰면
├── 브라우저가 원래 이해할 수 있는 문법이라
└── 개발 모드에서도 네이티브로 동작 (바꿔치기가 덜 필요)
```

### Vite 내장 변수

Vite는 `import.meta.env`에 몇 가지 기본 변수를 제공한다.

```
import.meta.env.MODE      → "development" | "production" (현재 모드)
import.meta.env.BASE_URL  → "/" (base 설정값)
import.meta.env.DEV       → true (개발 모드일 때)
import.meta.env.PROD      → true (프로덕션 모드일 때)
import.meta.env.SSR       → true (서버 사이드 렌더링일 때)
```

### MODE vs NODE_ENV

Vite에서 주의할 점이 하나 있다.
**MODE와 NODE_ENV는 다른 개념**이다.

```bash
vite dev                    → MODE="development", NODE_ENV="development"
vite build                  → MODE="production",  NODE_ENV="production"
vite build --mode staging   → MODE="staging",     NODE_ENV="production"
```

`--mode staging`으로 빌드하면 MODE는 `"staging"`이지만
NODE_ENV는 여전히 `"production"`이다.
`.env.staging` 파일은 MODE를 기준으로 로드된다.

### Vite 정리

```bash
1. .env 자동 로딩            #✅ (내장, dotenv 기반)
2. NODE_ENV 자동 설정        #✅
3. 클라이언트 노출 접두사       #VITE_
4. 환경 변수 접근 방식         #import.meta.env.* (process.env 아님!)
```

## Next.js — 서버와 클라이언트를 동시에 다루는 경우

Next.js는 **서버 사이드 렌더링(SSR) + 클라이언트 사이드**를 모두 처리하기 때문에
환경 변수 접근 방식이 가장 복잡하다.

### 두 개의 세계

```
서버 (getServerSideProps, API Routes, Server Components)
├── process.env.* 전부 접근 가능
├── DB 비밀번호, API 키 등 시크릿 사용 가능
└── 런타임에 값이 결정됨

클라이언트 (브라우저에서 실행되는 코드)
├── NEXT_PUBLIC_ 접두사가 붙은 것만 접근 가능
├── 빌드 타임에 문자열로 치환됨
└── 빌드 후에는 값 변경 불가
```

```tsx
// app/page.tsx (Server Component)
export default function Home() {
  // ✅ 서버 컴포넌트에서는 모든 환경 변수 접근 가능
  const dbUrl = process.env.DATABASE_URL          // ✅ 서버에서만 접근
  const apiUrl = process.env.NEXT_PUBLIC_API_URL  // ✅ 이것도 당연히 접근 가능

  return <ClientComponent />
}
```

```tsx
// components/ClientComponent.tsx
'use client'

export default function ClientComponent() {
  // ✅ NEXT_PUBLIC_ 접두사 → 클라이언트에서 접근 가능
  const apiUrl = process.env.NEXT_PUBLIC_API_URL  // "https://api.example.com"

  // ❌ 접두사 없음 → 클라이언트에서 undefined
  const dbUrl = process.env.DATABASE_URL  // undefined (빌드 시 치환 안 됨)

  return <div>API: {apiUrl}</div>
}
```

### NEXT*PUBLIC* 접두사의 동작 원리

CRA와 마찬가지로 **빌드 타임 문자열 치환**이다.

```js
// 빌드 전 (내가 쓴 코드)
const url = process.env.NEXT_PUBLIC_API_URL

// 빌드 후 (클라이언트 번들에 들어가는 코드)
const url = "https://api.example.com"
```

그래서 빌드 후에 `NEXT_PUBLIC_API_URL` 값을 바꿔도 반영이 안 된다.
이미 문자열이 박혀버렸기 때문이다.

### Next.js가 자동 설정하는 변수들

```
NODE_ENV
├── next dev   → "development"
├── next build → "production"
├── next start → "production"
└── 직접 설정할 필요 없음

PORT
├── 기본값: 3000
├── next dev -p 4000 또는 PORT=4000 으로 변경
└── 개발 서버 포트

NEXT_RUNTIME
├── "nodejs" 또는 "edge"
└── 현재 코드가 어디서 실행 중인지 표시
```

`NEXT_PRIVATE_*`, `__NEXT_*` 같은 변수들도 보이지만
이건 Next.js 내부 동작용이라 건드릴 일이 없다.

### .env 파일 자동 로딩

Next.js는 dotenv를 내장하고 있어서 `.env` 파일을 자동으로 읽어준다.
Express처럼 dotenv를 따로 설치할 필요가 없다.

```
NODE_ENV = "production"일 때 (next build)
1. .env                       ← 모든 환경 공통
2. .env.local                 ← 로컬 오버라이드 (gitignore 대상)
3. .env.production            ← production 전용
4. .env.production.local      ← production 로컬 오버라이드

4개 전부 다 읽고, 같은 변수는 나중에 읽힌 게 덮어씀
```

### Next.js 정리

```bash
1. .env 자동 로딩          #✅ (내장)
2. NODE_ENV 자동 설정      #✅
3. 클라이언트 노출 접두사     #NEXT_PUBLIC_
4. 환경 변수 접근 방식       #process.env.* (서버: 전부, 클라이언트: NEXT_PUBLIC_만)
```

## Webpack (직접 구성) — 전부 본인이 세팅

Webpack을 프레임워크 없이 직접 구성하면 **아무것도 자동으로 안 된다.**
`.env` 파일도 안 읽히고, `process.env`도 클라이언트에 노출 안 된다.

### 클라이언트에서 환경 변수를 쓰려면

Webpack의 `DefinePlugin` 또는 `EnvironmentPlugin`으로 직접 설정해야 한다.

```js
// webpack.config.js
const webpack = require('webpack')

module.exports = {
  plugins: [
    // 방법 1: DefinePlugin — 직접 치환할 값 지정
    new webpack.DefinePlugin({
      'process.env.NODE_ENV': JSON.stringify(process.env.NODE_ENV),
      'process.env.API_URL': JSON.stringify(process.env.API_URL),
    }),

    // 방법 2: EnvironmentPlugin — process.env에서 자동으로 가져옴
    new webpack.EnvironmentPlugin(['NODE_ENV', 'API_URL']),
  ],
}
```

### .env 파일을 읽으려면

```bash
npm install dotenv-webpack
```

```js
// webpack.config.js
const Dotenv = require('dotenv-webpack')

module.exports = {
  plugins: [
    new Dotenv({
      path: `./.env.${process.env.NODE_ENV}`,
    }),
  ],
}
```

### Webpack 정리

```bash
1. .env 자동 로딩            #❌ (dotenv-webpack 필요)
2. NODE_ENV 자동 설정        #❌ (직접 설정)
3. 클라이언트 노출 접두사       #없음 (직접 지정)
4. 환경 변수 접근 방식         #DefinePlugin으로 치환 설정 필요
```

## 전체 비교

|                    | .env 자동 로딩 | NODE_ENV 자동 설정 | 클라이언트 접근 방식 | 클라이언트 접두사 |
| ------------------ | :------------: | :----------------: | :------------------: | :---------------: |
| **Express**        |       ❌       |         ❌         |      해당 없음       |     해당 없음     |
| **CRA**            |       ✅       |         ✅         |   `process.env.*`    |   `REACT_APP_`    |
| **Vite**           |       ✅       |         ✅         | `import.meta.env.*`  |      `VITE_`      |
| **Next.js**        |       ✅       |         ✅         |   `process.env.*`    |  `NEXT_PUBLIC_`   |
| **Webpack (직접)** |       ❌       |         ❌         |  DefinePlugin 필요   |     직접 설정     |

## 자주 하는 실수

### 1. 클라이언트에서 시크릿 접근하려고 하기

```tsx
// ❌ 클라이언트 코드에서 DB 비밀번호 접근 시도
'use client'
const db = process.env.DATABASE_URL  // undefined

// ✅ 서버 코드에서 접근
// app/api/users/route.ts
export async function GET() {
  const db = process.env.DATABASE_URL  // 정상 동작
}
```

접두사(`NEXT_PUBLIC_`, `REACT_APP_`, `VITE_`) 없는 변수는
클라이언트에서 절대 접근할 수 없다. **이건 보안을 위한 의도적인 설계다.**

### 2. 빌드 후에 환경 변수 바꾸기

```bash
# 빌드
NEXT_PUBLIC_API_URL=https://api.com next build

# 실행할 때 다른 값으로 바꾸기
NEXT_PUBLIC_API_URL=https://other.com next start
# ❌ 안 됨! 빌드할 때 이미 "https://api.com"으로 치환됨
```

`NEXT_PUBLIC_*`이나 `REACT_APP_*` 같은 클라이언트 변수는
**빌드 타임에 고정**된다. 바꾸려면 다시 빌드해야 한다.

서버 전용 변수(접두사 없는 것)는 런타임에 바꿀 수 있다.

### 3. Vite에서 process.env 쓰기

```tsx
// ❌ Vite 프로젝트에서 process.env 사용
console.log(process.env.VITE_API_URL)  // "process is not defined"

// ✅ import.meta.env 사용
console.log(import.meta.env.VITE_API_URL)  // 정상 동작
```

CRA에서 Vite로 마이그레이션할 때 가장 많이 하는 실수다.
`process.env.REACT_APP_*`을 전부 `import.meta.env.VITE_*`로 바꿔야 한다.

### 4. .env 파일에 따옴표 넣기

```bash
# ❌ 따옴표가 값에 포함됨
API_URL="https://api.example.com"
# process.env.API_URL → '"https://api.example.com"' (따옴표 포함!)

# ✅ 따옴표 없이 작성
API_URL=https://api.example.com
```

dotenv 기반 도구들은 대부분 따옴표를 알아서 처리하지만,
OS 환경 변수나 Docker 환경에서는 따옴표가 값에 포함될 수 있으니 주의해야 한다.

## 마치며

`process.env`를 이해하려면 **"이 코드가 어디서 실행되는가"** 를 먼저 생각해야 한다.

```
서버에서 실행되는 코드 (Express, API Routes, Server Components)
→ process.env.* 전부 접근 가능, 시크릿도 OK

브라우저에서 실행되는 코드 (React 컴포넌트, Client Components)
→ process 자체가 없음
→ 번들러가 빌드 타임에 치환해준 것만 접근 가능
→ 접두사 규칙을 지켜야 함 (NEXT_PUBLIC_, REACT_APP_, VITE_)
```

프레임워크마다 문법이 다른 것 같지만 **근본 원리는 하나**다.

> **브라우저에는 process가 없으니까, 빌드할 때 문자열로 바꿔치기해준다.**

이 원리만 알면 어떤 프레임워크를 만나도 환경 변수 관련 문제를 해결할 수 있다.
