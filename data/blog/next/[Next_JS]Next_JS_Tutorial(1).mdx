지난번에 웹 모니터링 프로젝트를 nextjs를 활용해 만들 경험이 있었는데, 사실 리엑트의 프레임워크정도로 생각하고 사용했다.
깊이 있는 지식이 아니기에, 회사 업무가 널널한 요즈음 nextjs를 좀더 심도있게 공부해보려고 한다.

## NEXT-JS

공식 문서에서는 nextjs를 이렇게 설명한다.

> Next.js는 풀스택 웹 애플리케이션 구축을 위한 React 프레임워크입니다.
> React 컴포넌트를 사용하여 사용자 인터페이스를 구축하고, Next.js를 사용하여 추가 기능 및 최적화를 구현할 수 있습니다.

[공식 문서](https://nextjs.org/docs)

[vercel](https://vercel.com/)에서 개발하고 유지보수하는 nextjs는
서버사이드 렌더링(SSR), 정적 사이트 생성(SSG), 이미지 최적화,
API 라우트, 앱라우팅 등 웹 앱 개발에 필요한 기본 기능을 제공하는 프레임 워크이다. <br /><br />
장점이 많은데, **_가장 큰 사용 이유_**는 서버사이드 렌더링과, 정적 사이트 생성을 통한 SEO(Search Engine Optimization)의 최적화,
빠른 초기 렌더링 속도가 아닐까싶다. <br /> <br />
**_장점이 있으면 단점도 있다._** nextjs를 사용하는 단점은 무엇이 있을까? 빌드/배포 속도 문제, 무의미한 API 라우팅 제공, Server/Client 컴포넌트 혼란,
패키지 용량, 커스텀화 난이도, 환경별 동작 차이 등이 있을것이다. <br /> <br />
실제로 나도 지난 프로젝트를 만들어보면서 `use client`의 난발과 SEO최적화의 이해 부족, 커스텀화의 복잡성때문에 고초를 겪은바가 있다.
REACT에 매몰되어있던 나에게 신선하게 충격적이었지만, 귀찮고 자잘한 검색의 연속,
이해도의 부족과 같은 일들이 벌어지며 NEXT-JS에 대한 심도있는 공부로 이끈것 같다.<br /> <br />
아무튼 장점과 단점은 아래와 같이 정리하고자 한다.

### 📌 장점

#### 1. SEO(검색 엔진 최적화)에 유리

- 기본 React는 SPA(Single Page Application)라서, 최초 로딩 시 HTML이 비어있고 JS 실행 후 내용이 채워짐 → SEO 불리.
- Next.js는 SSR(서버사이드 렌더링)과 SSG(정적 사이트 생성)을 지원해서 검색엔진이 읽을 수 있는 HTML을 미리 제공 → 검색 노출 유리.

#### 2. 빠른 초기 로딩

- 서버에서 이미 렌더링된 HTML을 내려주기 때문에, 브라우저가 처음에 빈 화면을 띄우지 않고 바로 컨텐츠가 보임.
- 사용자가 “앱이 빠르다”는 인상을 받음.

#### 3. Image 최적화 / Code Splitting 자동 지원

- `<Image />` 컴포넌트로 자동 최적화 (WebP 변환, lazy-loading)
- 페이지 단위 코드 분리 (code splitting)로 번들 최적화

#### 4. API 라우트 지원

- /pages/api나 /app/api 안에 함수를 작성하면, 간단한 서버 API를 바로 만들 수 있음.
- 별도의 백엔드 서버 없이도 풀스택 개발 가능.

#### 5. App Router

- React Server Component(RSC) 도입 → 클라이언트/서버 코드 분리가 쉬움
- layout.tsx, loading.tsx, error.tsx 같은 기본 패턴 내장 → 개발 생산성 향상

### 📌 단점

#### 1. 빌드 및 배포 복잡성

- 페이지 수가 많거나 정적 사이트(SSG)를 대량으로 뽑을 때 빌드 시간이 길어짐.
- 대규모 블로그나 쇼핑몰 같은 경우 수천~수만 페이지를 SSG하면 빌드가 엄청 느림.
- ISR(Incremental Static Regeneration)으로 완화할 수 있지만, 캐싱/재검증 정책이 추가로 필요.

#### 2. 백엔드 한계

- API 라우트를 지원하긴 하지만, 대규모 백엔드 서버를 대체할 수준은 아님.
- 결국 규모가 커지면 별도의 백엔드(Node/Spring 등) 필요.
- DB 연결이나 인증 로직을 Next.js API Route에서 직접 다루면 서버리스 환경에서는 cold start 문제도 발생할 수 있다.

#### 3. Server/Client 컴포넌트 혼란

- use client 붙여야 하는지, 안 붙여도 되는지 처음에는 햇갈림.
- 서버 컴포넌트에서는 useState, useEffect, 브라우저 API 사용 불가.
- 라이브러리(MUI, styled-components 등)와 SSR 호환 문제로 Hydration Error가 자주 발생.

#### 4. 패키지 용량

- CRA(Create React App) + Vite 같은 대안보다 무겁고 복잡함.
- “간단한 내부용 대시보드” 같은 경우에는 오히려 오버스펙이 될 수 있음.

#### 5. 커스텀화 난이도

- Next.js가 많은 걸 해주긴 하지만, 내부 동작이 추상화돼 있어서 세부 제어가 어려움.
- Webpack 커스터마이징, 라우팅 제어, 캐싱 동작을 바꾸려면 공식 문서를 깊이 이해해야함.

#### 6. 환경별 동작 차이

- Vercel을 위한 최적화가 많아서, AWS/GCP/자체 서버 등에 배포할 때는 설정이 번거롭운 경우 발생.

### 마치며

이번 글에서는 Next.js의 장점과 단점을 정리해봤다.  
내가 직접 프로젝트를 진행하며 느낀 경험까지 더해지니, 공식 문서만 읽을 때보다 훨씬 현실적인 장단점이 보였다.

Next.js는 분명히 강력하고 매력적인 프레임워크지만, 동시에 러닝커브와 환경 설정의 복잡함이 뒤따른다.  
즉, 무조건 “좋다/나쁘다”의 문제가 아니라 **"무엇을 만들 것인가”에 따라 선택이 달라지는 도구**라는 생각이 든다.

앞으로는 단순히 “써본다”에 그치지 않고, 왜 이렇게 동작하는지, 어떤 상황에서 더 적합한지 깊이 이해하면서 써야겠다.  
다음 글에서는 공식 문서를 더 파고들며, 실제로 적용 가능한 기능들을 하나씩 정리해볼 예정이다.
