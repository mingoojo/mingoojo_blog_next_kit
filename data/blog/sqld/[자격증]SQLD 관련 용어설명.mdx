### 스키마(schema)

> **데이터베이스 구조 또는 설계도를 의미**

- 스키마는 데이터베이스에서 사용될때, 테이블(엔터티)안에서 사용될때 의미가 달라진다.

1.  **데이터베이스의 스키마(Database Schema)**
    - 데이터베이스의 논리적인 구조를 나타낸다.
    - 데이터베이스 안에 테이블, 뷰, 인덱스, 관계. 제약조건, 사용자 권한등을 포함한 구조를 통틀어 말한다.

    - 구성 요소 예시
      - 테이블 구조
      - 테이블 간 관계 (1:N, N:M)
      - 제약조건 (PK, FK, UNIQUE, CHECK 등)
      - 인덱스, 뷰, 트리거
      - 사용자 권한과 접근 정책

    - 건물 전체의 설계도

2.  **테이블 스키마(Table Schema)**
    - 특정 테이블 하나의 구조를 정의 한 것
    - 테이블에 존재하는 열, 열의 데이터 타입, 제약조건 등을 의미

      ```sql
          CREATE TABLE users (
          user_id INT PRIMARY KEY,
          name VARCHAR(50) NOT NULL,
          email VARCHAR(100) UNIQUE,
          created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
          );
      ```

    - 건물 안 한 방(테이블) 의 내부 설계도

### 트랜젝션(Transaction)

> **데이터베이스의 논리적 작업단위**

- INSERT, UPDATE, DELETE와 같은 여러작업을 할때 한번에 처리하는 단위로 묶어서 취급

1. **트랜잭션의 특징**
   - 원자성 : 하나의 트랜잭션은 모두 실행하거나 모두 실행하지 않거나 둘중 하나만의 결과만 낸다.
     - 일부만 반영되는 일은 없다. 실패 시 전체가 롤백되어 이전 상태로 복구된다.
   - 일관성 : 트랜잭션이후와 이전 모두 정합성을 갖고 이전에 오류가 없다면 이후에도 오류가 없다.
     - 데이터의 제약조건이나 규칙이 항상 유지되어야 한다. (예: 계좌 잔액이 음수가 되지 않음)
   - 고립성 : 트랜잭션관의 영향은 없다.
     - 동시에 실행되더라도 서로 간섭하지 않고 독립적으로 처리된다.
   - 영속성 : 트랜잭션 결과값이 영구적으로 DB에 저장된다.
     - 커밋 후에는 전원 꺼짐이나 장애가 발생해도 데이터가 보존된다.

2. **트랜잭션의 격리수준이 낮을때 특성**
   - Dirty Read : 트랜잭션에 의해 수정된 후 커밋 되기전에 다른 트랜잭션에서 읽어가면서 생기는 오류
     - 아직 확정되지 않은 임시 데이터를 다른 트랜잭션이 읽어 잘못된 값을 참조하게 되는 현상.
   - Not Repeatable Read : 같은 트랜잭션을 반복할때 값이 바뀌어 결과가 달라짐
     - 두 번의 조회 사이에 다른 트랜잭션이 값을 수정하여, 한 번은 10, 다음엔 12가 되는 상황.
   - Phantom Read : 같은 트랜잭션이 두번 반복될때 조회결과 값의 갯수가 변경되는 것
     - 다른 트랜잭션이 데이터를 추가하거나 삭제해서, 재조회 시 결과 집합의 크기가 달라지는 현상.

> **따라서 트랜잭션은 신뢰성 있는 데이터 처리의 최소 단위라고 할 수 있다.**
