### 1. 기출문제 1회차

    1. **문제 1번**
      - ```
        다음 중 데이터베이스 논리 모델에 대한 설명으로 옳지 않은 것은? [답 : 3]

            ① 개념 데이터 모델은 사용자 관점에서 데이터 요구사항을 식별한다.
            ② 논리 데이터 모델은 개념 모델과 해소, 식별자 확정, 정규화, 무결성 정의 등을 수행한다.
            ③ 논리 모델은 데이터베이스 구축을 위해서 사용되는 것이다.
            ④ 데이터가 물리적으로 저장되는 방법을 정의하는 것이 물리적 모델이다.
            ```
      - 개념적 모델링(ERD만들기)
        - 데이터 베이스를 추상화 > 엔터티, 속성, 관계를 식별(정의)하는 과정
        - 사용자와 데이터 설계자간에 소통의 역할
        - 결과물 : `ERD`
      - 논리적 모델링(정규화)
        - 개념모델을 실제 DBMS에 구현가능한 형태로 변환
        - 주요작업 :
            - 정규화 (1NF~3NF)
            - 키 설정 (기본키, 외래키)
            - 관계(1:N, N:M 등) 구체화
        - 결과물 : 논리적 스키마
            - | 테이블명    | 컬럼명       | 설명   | 키                       |
                |---------|-----------|------|-------------------------|
                | STUDENT | STU_ID | 학번 | PK |
                | STUDENT | NAME | 이름 | |
                | STUDENT | DEPT | 학과 | |
                | COURSE | COURSE_ID | 과목코드 | PK |
                | COURSE | TITLE | 과목명 | |
                | COURSE | CREDIT | 학점 | |
                | ENROLL | STU_ID | 학번 | FK (→ STUDENT.STU_ID) |
                | ENROLL | COURSE_ID | 과목코드 | FK (→ COURSE.COURSE_ID) |
                | ENROLL | GRADE | 성적 |
      - 물리적 모델링(DDL생성)
        - 실제 DBMS의 특징과 성능을 고려한 설계
        - 주요 작업
            - 인덱스 설계
            - 파티션 구성
            - 테이블스페이스 및 스토리지 설계
            - DBMS별 자료형 매핑
        - 결과물 : DDL
            - ```sql
                -- 학생 테이블
                CREATE TABLE STUDENT (
                    STU_ID CHAR(8) PRIMARY KEY,     -- 학번
                    NAME VARCHAR(50) NOT NULL,      -- 이름
                    DEPT VARCHAR(30)                -- 학과
                );

                -- 과목 테이블
                CREATE TABLE COURSE (
                    COURSE_ID CHAR(6) PRIMARY KEY,  -- 과목코드
                    TITLE VARCHAR(100) NOT NULL,    -- 과목명
                    CREDIT INT CHECK (CREDIT BETWEEN 1 AND 5)  -- 학점 제약조건
                );

                -- 수강 테이블 (학생과 과목의 관계)
                CREATE TABLE ENROLL (
                    STU_ID CHAR(8),                 -- 학번
                    COURSE_ID CHAR(6),              -- 과목코드
                    GRADE CHAR(1),                  -- 성적 (A/B/C/D/F)
                    PRIMARY KEY (STU_ID, COURSE_ID), -- 복합 기본키
                    FOREIGN KEY (STU_ID) REFERENCES STUDENT(STU_ID),
                    FOREIGN KEY (COURSE_ID) REFERENCES COURSE(COURSE_ID)
                );

            ```

    2. **문제 2번**
      - ```
        02. 다음에서 설명하는 것은 ER모델 중 어떤 항목에 대한 설명인가? [ 답: 2 ]

                가) 모든 릴레이션(Relation)은 원자값(Atomic)을 가져야 한다.
                나) 어떤 릴레이션(Relation)에서 속성값이 가질 수 있는 값의 범위를 의미한다.
                다) 실제 속성이 올바르게 표시되었는지 확인한다.
                라) 속성명은 반드시 동일할 필요는 없다.

                ① 카디널리티(Cardinality)
                ② 도메인(Domain)
                ③ 인스턴스(Instance)
                ④ 차수(Degree)
        ```
      - 질문의 의미
        - ER모델 = ERD와 엔터티가 정의된 상황의 엔터티
        - 다음에서 설명하는 것은 ER모델 중 어떤 항목에 대한 설명인가? = ERD 안에 있는 구성요소(=엔터티, 속성, 관계 등) 중 어떤 걸 설명하는 문장인가?
      - 질문 설명
        - 가) 모든 릴레이션(Relation)은 원자값(Atomic)을 가져야 한다.
            - 모든 릴레이션(테이블, 엔터티)는 원자값을 가져야 한다 > 도메인의 설정시 조건
            - 모든 릴레이션의 값(속성값)은 원자값을 가져야 한다 > 속성값의 복수성을 허락하지 않는다.
        - 나) 어떤 릴레이션(Relation)에서 속성값이 가질 수 있는 값의 범위를 의미한다.
            - 도메인의 뜻을 의미함
        - 다) 실제 속성이 올바르게 표시되었는지 확인한다.
            - 속성값이 도메인에 정의된 범위 내에 있는지 검사한다. > 도메인의 제약조건을 의미
        - 라) 속성명은 반드시 동일할 필요는 없다.
            - 도메인은 속성명과 무관하게 재사용 가능
      - 항목 설명
        - 카디널리티 : 개체(Entity) 간 관계(relationship)의 수적 대응 관계를 의미(관계 차수와 같은 의미)
        - 도메인 : 속성에 속성값이 가질 수 있는 값의 범위
        - 인스턴스 : 릴레이션(테이블, 엔터티)가 같는 값> 레코드, 튜플 = 테이블의 한 행(row)
        - 차수 : 하나의 릴레이션(테이블)에 존재하는 속성(Attribute)의 개수
      - ER모델 특징 설명
        - | 구분                                  | 이름                          | 설명                    | 예시                                           |
            |-------------------------------------|-----------------------------|-----------------------|----------------------------------------------|
            | 1️⃣ Entity (개체)                     | 현실 세계의 독립적인 객체              | 사람, 과목, 주문 등          | 학생(Student)                                  |
            | 2️⃣ Attribute (속성)                  | 개체가 가진 구체적인 특성              | 학번, 이름, 학과 등          | 이름='홍길동'                                     |
            | 3️⃣ Key (키)                         | 각 엔터티의 인스턴스를 유일하게 식별하는 속성   | 기본키, 후보키, 대체키 등       | 학번, 주민번호 등                                   |
            | 4️⃣ Relationship (관계)               | 개체 간의 연관성                   | 학생이 과목을 수강한다          | 학생-수강-과목                                     |
            | 5️⃣ Cardinality (카디널리티)             | 관계의 수적 비율                   | 1:1, 1:N, N:M         | 고객:주문 = 1:N                                  |
            | 6️⃣ Domain (도메인)                    | 속성값이 가질 수 있는 값의 집합(범위)      | 성별=\{남,여}              | 학점=\{A,B,C,D,F}                               |
            | 7️⃣ Instance (인스턴스)                 | 실제 데이터 값 (테이블의 한 행)         | 튜플(row)               | ('S001', '홍길동', '컴퓨터공학과')                    |
            | 8️⃣ Degree (차수)                     | 릴레이션의 속성(컬럼) 수              | STUDENT의 컬럼이 5개면 차수=5 | 5                                            |
            | 9️⃣ Participation (참여도)             | 개체가 관계에 참여하는 필수 여부          | 전체참여 / 부분참여           | 학생이 수강을 반드시 해야 한다면 ‘전체참여’                    |
            | 🔟 Identifying Relationship (식별 관계) | 자식 엔터티의 기본키가 부모의 키를 포함하는 관계 | 강의실(건물) → 강의(건물,호실)   | 식별 관계(Identifying) / 비식별 관계(Non-Identifying) |

    3. **문제 3번**
      - ```
        03. 속성에 대한 아래의 설명에서 빈칸에 들어갈 것으로 올바른 것은?

            (ㄱ) _____ 은(는) 엔터티를 식별할 수 있는 속성이며,
            (ㄴ) _____ 은(는) 다른 엔터티의 관계에 포함되는 속성이다.
            다른 엔터티의 관계에 포함되지 않는 속성을 (ㄷ) _____ 이라고 한다.

            ① 기본키 속성 – 외래키 속성 – 일반 속성
            ② 외래키 속성 – 기본키 속성 – 파생 속성
            ③ 파생 속성 – 외래키 속성 – 기본키 속성
            ④ 일반 속성 – 기본키 속성 – 외래키 속성
        ```
      - 설명
        - 기본키 속성 (Primary Key Attribute) : 엔터티를 유일하게 식별할 수 있는 속성
        - 외래키 속성 (Foreign Key Attribute) : 다른 엔터티와의 관계에 포함되어 연결을 표현하는 속성
        - 일반 속성 (Non-key Attribute) : 식별이나 관계와는 무관한, 일반적인 정보 속성

    5. **문제 5번**
      - ```
        05. 다음 보기 중 슈퍼타입/서브타입 데이터 모델의 변환단계에 대한 설명으로 옳은 것은?

            ① One To One(1:1) 개념으로 발생되는 트랜잭션에 대해서는
               개별 테이블로 구성하는 것으로 테이블의 수가 많아진다.
            ② Plus Type은 하나의 테이블을 생성하는 것으로 조인(Join)이 발생하지 않는다.
            ③ Two Type은 슈퍼와 서브타입 형식으로 데이터를 처리하는 경우로
               조인성이 우수하여 Super Type과 Sub Type 변환 시에 항상 사용된다.
            ④ One To One Type은 조인성이 우수하기 때문에 관리가 편리하다.
        ```
      - 설명
        - 슈퍼타입 / 서브타입이란?
          - ER 모델(개념적 모델링) 에서 현실 세계의 개체(Entity)들이 공통점은 같지만, 일부 특징이 다를 때 그 관계를 “상속 구조”처럼 표현하는 개념.
          - | 구분               | 의미                          | 비유       | 예시           |
            |------------------|-----------------------------|----------|--------------|
            | 슈퍼타입 (Supertype) | 여러 개체의 공통 속성을 모아둔 상위 개체     | “부모 클래스” | 직원(Employee) |
            | 서브타입 (Subtype)   | 슈퍼타입을 세분화한 하위 개체, 고유 속성을 가짐 |
          - 등장배경
            - 현실에선 비슷한 데이터가 많지만, 그걸 전부 따로 관리하면 중복이 너무 많아지고 비효율적이기에 효율적인 관리를 위해 등장.
            - One-To-One Type / Plus Type / Two Type으로 나누어서 구분됨
            - | 방식                  | 구조                        | 특징                    |
              | ------------------- | ------------------------- | --------------------- |
              | **One-To-One Type** | 슈퍼타입 + 모든 서브타입 각각 테이블로 분리 | 완전 정규화 / 조인 많음        |
              | **Plus Type**       | 하나의 테이블에 전부 저장            | 단순 / NULL 많음          |
              | **Two Type**        | 슈퍼타입 + 일부 서브타입만 분리        | 현실적인 절충형 (실무에서 자주 사용) |

          - 예시
            - | 사번   | 이름    | 입사일        | 고용형태 | 연봉<br/>(정규직에만 적용)   | 계약기간<br/>(계약직에만 적용) | 학교명<br/>(인턴에만 적용)   |
              | ---- | --- | ---------- | ---- | ---- | ---- | ----- |
              | E001 | 김민수  | 2022 | 정규직  | 5000 | NULL | NULL  |
              | E002 | 박지현  | 2023 | 계약직  | NULL | 12   | NULL  |
              | E003 | 이서준  | 2023 | 인턴   | NULL | NULL | 서울대학교 |
              - 사번, 이름, 부서, 입사일 → 공통 속성 (슈퍼타입)
              - 연봉, 계약기간, 학교명 → 서브타입 전용 속성(서브타입)
            - | 방식             | 구조                                              | 인스턴스 예시                     |
              | -------------- | ----------------------------------------------- | --------------------------- |
              | **One-To-One** | EMPLOYEE + PERM_EMP + CONTRACT_EMP + INTERN_EMP | 모든 유형이 각자 테이블에 있음           |
              | **Two Type**   | EMPLOYEE + CONTRACT_EMP                         | 계약직만 별도, 나머진 EMPLOYEE에      |
              | **Plus Type**  | EMPLOYEE 1개 테이블                                 | NULL 많음 (정규직은 계약기간이 NULL 등) |
