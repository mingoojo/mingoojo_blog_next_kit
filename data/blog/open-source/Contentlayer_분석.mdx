## Contentlayer가 뭔지부터

[공식문서](https://contentlayer.dev/docs/getting-started-cddd76b7)

한 줄로 표현하면 **"MDX/Markdown 파일을 타입 안전한 JSON 데이터로 변환해주는 빌드 타임 SDK"** 다.

처음 들으면 "그냥 `fs.readFileSync`로 읽으면 되는 거 아냐?" 라고 생각할 수 있는데,
직접 파일을 읽는 방식과 Contentlayer를 쓰는 방식의 차이를 보면 바로 이해된다.

```ts
  // ❌ 직접 파일 읽기 방식
  import fs from 'fs'
  import matter from 'gray-matter'

  const files = fs.readdirSync('posts')
  const posts = files.map((file) => {
    const raw = fs.readFileSync(`posts/${file}`, 'utf-8')
    const { data, content } = matter(raw)
    return { ...data, content }  // 타입? any. slug? 직접 계산. 검증? 없음.
  })

  // ✅ Contentlayer 방식
  import { allBlogs } from 'contentlayer/generated'

  // allBlogs는 이미 타입 완전히 보장됨
  // slug, readingTime, toc 전부 자동 계산됨
  // required 필드 없으면 빌드에서 미리 에러
```

## contentlayer vs contentlayer2 — 왜 fork가 생겼나

원래 `contentlayer`는 **Stackbit**이라는 회사가 만들고 유지보수했다.
근데 2023년 중반 Netlify가 Stackbit을 인수하면서 contentlayer 개발이 사실상 멈췄다.

마지막 공식 릴리즈는 0.3.4인데, 이 시점 이후로 Next.js 14, 15, React 19 등
주요 의존성들이 계속 업데이트되면서 호환 문제가 쌓이기 시작했다.

그래서 `timlrx` (tailwind-nextjs-starter-blog 제작자)가 커뮤니티 포크를 만들었는데,
그게 바로 **contentlayer2**다.

### contentlayer2가 contentlayer와 다른 점

```
contentlayer (원본, 중단됨)
├── Unified v9 기반 (구버전)
├── Next.js 13까지만 안정적으로 지원
└── TypeScript 4.x 지원

contentlayer2 (커뮤니티 포크, 유지보수 중)
├── Unified v11+ 기반 (최신)
├── Next.js 15까지 지원
├── TypeScript 5+ 요구 (ts-pattern 최신 버전 때문)
└── API는 100% 동일 — 패키지 이름만 다름
```

마이그레이션은 패키지 이름만 바꾸면 된다.

```bash
# before
contentlayer, next-contentlayer

# after
contentlayer2, next-contentlayer2
```

```ts
// before
import { defineDocumentType, makeSource } from 'contentlayer/source-files'

// after
import { defineDocumentType, makeSource } from 'contentlayer2/source-files'
```

## 내부 동작 원리

Contentlayer2가 어떻게 동작하는지 단계별로 파헤쳐봤다.

### 전체 파이프라인

```
1. 파일 감지
   contentlayer.config.ts의 contentDirPath 스캔
   filePathPattern과 매칭되는 파일 수집

2. 콘텐츠 파싱
   gray-matter로 frontmatter 분리
   남은 본문은 MDX/Markdown으로 처리

3. 스키마 검증
   defineDocumentType에서 선언한 fields와 대조
   required 필드 없으면 빌드 에러 발생

4. computedFields 실행
   각 필드의 resolve 함수 실행
   slug, readingTime, toc 등 자동 생성

5. MDX/Markdown 컴파일
   remark 플러그인 체인 실행 (Markdown → MDAST)
   rehype 플러그인 체인 실행 (MDAST → HTML/JSX)

6. 결과물 생성
   .contentlayer/generated/ 에 JSON 파일 생성
   TypeScript 타입 정의 파일 자동 생성
   index.mjs export 파일 생성
```

### 생성되는 파일 구조

```
.contentlayer/generated/
├── Blog/
│   ├── _index.json           ← 모든 블로그 글 데이터
│   ├── hello-world.mdx.json  ← 개별 글 데이터
│   └── series__part-1.mdx.json
├── Authors/
│   └── _index.json
├── index.d.ts                ← 자동 생성된 TypeScript 타입
└── index.mjs                 ← ESM export
```

`index.d.ts`를 열어보면 이런 식으로 타입이 자동 생성되어 있다.

```ts
// .contentlayer/generated/index.d.ts (자동 생성)
export interface Blog {
  title: string
  date: IsoDateTimeString
  tags: string[]
  draft?: boolean
  summary?: string
  // computedFields
  slug: string
  readingTime: ReadingTime
  toc: TocItem[]
  body: MDX
  _id: string
  _raw: RawDocumentData
}

export declare const allBlogs: Blog[]
```

이게 빌드할 때마다 새로 생성되기 때문에 타입이 항상 실제 파일과 일치한다.

## defineDocumentType 완전 분석(contentlayer.config.ts)

### defineDocumentType이 하는 일

`defineDocumentType`으로 타입을 정의하면 contentlayer가 빌드 타임에 3가지를 자동으로 만들어준다.

1. TypeScript 타입 생성

   ```ts
   // .contentlayer/generated/index.d.ts (자동 생성)
   export interface Blog {
      title: string
      date: IsoDateTimeString
      tags: string[]
      slug: string        // computedField
      readingTime: JSON   // computedField
      body: MDX
   }
   ```

2. allXxx 배열 생성

   ```ts
   export declare const allBlogs: Blog[]
   export declare const allAuthors: Authors[]
   ```

3. import해서 바로 사용 가능

   ```ts
   import { allBlogs } from 'contentlayer/generated'
   import type { Blog } from 'contentlayer/generated'

   // 완전한 타입 보장 상태로 사용
   const post: Blog = allBlogs[0]
   post.title        // string
   post.readingTime  // JSON
   ```

   즉 `defineDocumentType` 하나가 **스키마 정의 → 타입 생성 → 데이터 접근**을 전부 연결해준다.

### defineDocumentType 구조

```ts
//contentlayer.config.ts
import { defineDocumentType } from 'contentlayer2/source-files'

export const Blog = defineDocumentType(() => ({
  name: 'Blog',               // 타입명 — allBlogs, Blog 타입 이름이 됨
  filePathPattern: 'blog/**/*.mdx',  // glob 패턴으로 파일 매칭
  contentType: 'mdx',         // 'mdx' | 'markdown' | 'data'
  fields: { ... },
  computedFields: { ... },
}))
```

1.  contentType 설정

    ```ts
    // mdx: JSX 포함 가능, 컴포넌트 삽입 가능
    contentType: 'mdx'

    // markdown: 순수 마크다운만. 더 빠른 처리
    contentType: 'markdown'

    // data: YAML, JSON 파일용. body 없음
    contentType: 'data'
    ```

    블로그 글처럼 MDX 컴포넌트를 쓰려면 `mdx`,
    순수 마크다운 문서라면 `markdown`이 더 빠르다.

2.  fields — 타입 시스템

    지원하는 필드 타입 전체 목록이다.

    ```ts
      fields: {
         // 내가 원하는 필드 뭐든 추가 가능
         // 기본 타입들
         title:    { type: 'string',  required: true },
         count:    { type: 'number' },
         isPublic: { type: 'boolean' },
         date:     { type: 'date' },     // Date 객체로 자동 파싱
         // 리스트
         tags: {
            type: 'list',
            of: { type: 'string' },
            default: []
         },
         // enum — 정해진 값만 허용
         status: {
            type: 'enum',
            options: ['draft', 'published', 'archived'],
            default: 'draft',
         },
         // json — 자유 형태 객체
         metadata: { type: 'json' },
      }
    ```

3.  defineNestedType — 재사용 가능한 중첩 타입

    필드 안에 구조화된 객체가 필요할 때 쓴다.

    ```ts
       import { defineNestedType, defineDocumentType } from 'contentlayer2/source-files'

       // 재사용 가능한 중첩 타입 정의
       const SEO = defineNestedType(() => ({
          name: 'SEO',
          fields: {
          title: { type: 'string' },
          description: { type: 'string' },
          image: { type: 'string' },
          },
       }))

       const Series = defineNestedType(() => ({
          name: 'Series',
          fields: {
          title: { type: 'string', required: true },
          order: { type: 'number', required: true },
          },
       }))

       export const Blog = defineDocumentType(() => ({
          name: 'Blog',
          filePathPattern: 'blog/**/*.mdx',
          contentType: 'mdx',
          fields: {
             title: { type: 'string', required: true },
             seo:    { type: 'nested', of: SEO },      // 중첩 객체
             series: { type: 'nested', of: Series },   // 시리즈 정보
          },
       }))
    ```

4.  isSingleton — 단일 파일 타입
    사이트 전체 설정처럼 딱 하나만 있는 파일에 쓴다.

    ```ts
    export const SiteConfig = defineDocumentType(() => ({
      name: 'SiteConfig',
      filePathPattern: 'config/site.yaml',
      isSingleton: true,   // allSiteConfigs 대신 siteConfig로 단일 export
      contentType: 'data',
      fields: {
         title:       { type: 'string', required: true },
         description: { type: 'string' },
         url:         { type: 'string', required: true },
      },
    }))
    ```

    ```ts
      // 사용할 때
      import { siteConfig } from 'contentlayer/generated'  // 배열이 아닌 단일 객체
    ```

## computedFields

1. 기본 — slug와 path

   ```ts
    computedFields: {
    // 파일 경로에서 slug 추출
    // blog/my-post.mdx          → "my-post"
    // blog/series/part-1.mdx    → "series/part-1"
    slug: {
       type: 'string',
       resolve: (doc) => doc._raw.flattenedPath.replace(/^.+?(\/)/, ''),
    },

    // 페이지 URL
    path: {
       type: 'string',
       resolve: (doc) => `/blog/${doc._raw.flattenedPath.replace(/^.+?(\/)/, '')}`,
    },
    }
   ```

2. \_raw 객체가 뭔가

   `doc._raw`는 파일에 대한 메타 정보를 담고 있다.

   ```ts
   doc._raw = {
      sourceFilePath: 'blog/series/part-1.mdx',   // 전체 경로
      sourceFileName: 'part-1.mdx',               // 파일명만
      sourceFileDir: 'blog/series',               // 디렉토리
      flattenedPath: 'blog/series/part-1',        // 확장자 제거한 전체 경로
      contentType: 'mdx',
   }
   ```

   slug를 파일명 기준으로 만들고 싶으면:

   ```ts
   slug: {
   type: 'string',
   resolve: (doc) => doc._raw.sourceFileName.replace(/\.mdx$/, ''),
   // blog/series/part-1.mdx → "part-1" (디렉토리 무시)
   }
   ```

   경로 포함 slug가 필요하면:

   ```ts
   slug: {
   type: 'string',
   resolve: (doc) => doc._raw.flattenedPath.replace(/^blog\//, ''),
   // blog/series/part-1 → "series/part-1"
   }
   ```

3. readingTime 계산

   ```ts
   import readingTime from 'reading-time'

   readingTime: {
   type: 'json',
   resolve: (doc) => readingTime(doc.body.raw),
   // 반환값: { text: '5 min read', minutes: 5, time: 300000, words: 1000 }
   }
   ```

## makeSource 완전 분석

`makeSource`는 `contentlayer.config.ts`의 최종 조립 함수이다.
`defineDocumentType`으로 타입을 정의하고, `makeSource`로 그걸 전부 묶어서 "이제 이 설정대로 빌드해" 라고 `contentlayer에` 전달.

1. 핵심 옵션들

   ```ts
   export default makeSource({
      // 콘텐츠 루트 디렉토리
      contentDirPath: 'data',

      // 특정 디렉토리만 포함
      contentDirInclude: ['blog', 'authors'],

      // 제외할 디렉토리
      contentDirExclude: ['drafts', 'templates'],

      // 등록할 문서 타입들
      documentTypes: [Blog, Authors],

      // MDX 파이프라인 설정
      mdx: {
         remarkPlugins: [...],
         rehypePlugins: [...],
      },

      // 빌드 완료 후 실행할 콜백
      onSuccess: async (importData) => { ... },

      // 필드 옵션
      fieldOptions: {
         bodyFieldName: 'content',    // 기본값: 'body'
         typeFieldName: '__typename', // 기본값: 'type'
      },

      // 날짜 파싱 옵션
      date: {
         timezone: 'Asia/Seoul',  // 타임존 지정
      },
   })
   ```

## 마치며

Contentlayer2를 쓰는 이유를 한마디로 정리하면:

> **"마크다운 파일을 DB처럼 쓰면서, 타입 안전성까지 가져가는 방법"**

원본 contentlayer 공식 문서가 아직 살아있으니 (`contentlayer.dev`) 필드 타입이나
고급 설정이 필요하면 참고하면 된다. API가 contentlayer2랑 동일하다.
