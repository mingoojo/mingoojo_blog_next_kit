## 1. 프로젝트 생성과 의존성 설치

### 프로젝트 생성

```bash
npx create-next-app@latest my-blog \
  --typescript \
  --tailwind \
  --eslint \
  --app \
  --src-dir \
  --import-alias "@/*"
```

`--src-dir` 옵션으로 생성하면 `src/app/` 구조가 만들어진다.
나중에 `@/` 경로 설정이 `src/` 기준인지 루트 기준인지 헷갈릴 수 있으니 처음에 잘 잡아야 한다.

### 의존성 설치

패키지가 많은데 역할별로 나눠서 이해하면 쉽다.

```bash
# Contentlayer - MDX 파일을 타입 안전한 JSON으로 변환
npm install contentlayer2 next-contentlayer2

# MDX 파이프라인 - 마크다운 → HTML 변환 플러그인들
npm install \
  remark remark-gfm remark-math \
  rehype-slug rehype-autolink-headings \
  rehype-prism-plus rehype-katex \
  unist-util-visit hast-util-from-html-isomorphic

# UI / 유틸
npm install next-themes pliny reading-time github-slugger gray-matter

# Tailwind 플러그인
npm install @tailwindcss/typography @tailwindcss/forms

# 개발 도구
npm install -D husky cross-env @next/bundle-analyzer concurrently
```

각 플러그인이 뭘 하는지 간단히 정리하면:

- `remark-gfm` — GitHub Flavored Markdown. 테이블, 체크박스, 취소선 지원
- `remark-math` — `$E=mc^2$` 같은 수식 문법 인식
- `rehype-slug` — 헤딩에 id 자동 부여 (`<h2 id="시작하며">`)
- `rehype-autolink-headings` — 헤딩에 앵커 링크 자동 추가
- `rehype-prism-plus` — 코드 블록 문법 하이라이팅
- `rehype-katex` — 수식을 실제 HTML로 렌더링
- `reading-time` — 글 읽는 데 걸리는 예상 시간 자동 계산
- `github-slugger` — `My Post Title` → `my-post-title` 변환
- `pliny` — 블로그에 필요한 유틸 함수 모음 (sortPosts 등)

## 2. 버전 이슈 — Next.js 15로 다운그레이드

### 문제

`create-next-app` 최신 버전으로 생성하면 **Next.js 16**이 설치된다.
근데 `next-contentlayer2`가 아직 Next.js 16 + Turbopack을 완전히 지원하지 않아서
개발 서버를 켜면 이런 에러가 난다.

```
ERROR: This build is using Turbopack, with a `webpack` config and no `turbopack` config.
```

`next-contentlayer2`가 webpack 플러그인 방식으로 Next.js 빌드 과정에 끼어드는데,
Turbopack이 기본값이 된 Next.js 16에서는 이 방식이 제대로 동작하지 않는 거다.

### 해결

원본 템플릿과 동일하게 **Next.js 15로 다운그레이드**.

```bash
npm install next@15.0.2 react@rc react-dom@rc
```

```json
// package.json
{
  "scripts": {
    "dev": "cross-env INIT_CWD=$PWD next dev",
    "build": "cross-env INIT_CWD=$PWD next build"
  }
}
```

Next.js 15에서는 `withContentlayer`가 webpack 플러그인으로 빌드에 자동으로 끼어들어서
별도로 contentlayer를 실행하지 않아도 `npm run dev` 하나로 다 된다.

## 3. contentlayer.config.ts — 핵심 엔진

이 파일이 프로젝트에서 가장 중요하다. 역할이 세 가지다.

```
1. 문서 타입 정의  →  Blog 글은 어떤 frontmatter를 가지는가
2. computedFields  →  빌드 타임에 자동으로 계산할 값들
3. MDX 파이프라인  →  마크다운을 HTML로 변환할 때 어떤 플러그인을 쓸건가
```

### 전체 데이터 흐름

```
data/blog/my-post.mdx
  ↓
contentlayer (빌드 타임 실행)
  ├── frontmatter 파싱 + 타입 검증
  ├── computedFields 계산 (slug, readingTime, toc...)
  ├── remark 플러그인 체인 (마크다운 → AST)
  └── rehype 플러그인 체인 (AST → HTML)
  ↓
.contentlayer/generated/Blog/_index.json
  ↓
import { allBlogs } from 'contentlayer/generated'
// 완전한 TypeScript 타입 보장
```

### computedFields

frontmatter에 직접 쓰지 않아도 빌드 타임에 자동으로 계산되는 필드들이다.
이게 contentlayer의 핵심 기능 중 하나.

```ts
const computedFields: ComputedFields = {
  // 파일 경로에서 slug 자동 추출
  // data/blog/my-post.mdx          → "my-post"
  // data/blog/series/part-1.mdx    → "series/part-1"
  slug: {
    type: 'string',
    resolve: (doc) => doc._raw.flattenedPath.replace(/^.+?(\/)/, ''),
  },

  // 글 읽는 시간 자동 계산
  // { text: '5 min read', minutes: 5, words: 1000 }
  readingTime: {
    type: 'json',
    resolve: (doc) => readingTime(doc.body.raw),
  },

  // 페이지 URL 자동 생성
  path: {
    type: 'string',
    resolve: (doc) => `/${doc._raw.flattenedPath}`,
  },

  // 헤딩 추출 → 목차(TOC) 자동 생성
  toc: {
    type: 'json',
    resolve: (doc) => {
      const slugger = new GithubSlugger()
      const headingLines = doc.body.raw
        .split('\n')
        .filter((line: string) => line.match(/^#{1,6}\s/))

      return headingLines.map((raw: string) => {
        const text = raw.replace(/^#{1,6}\s/, '')
        // #의 개수로 레벨 계산 (h1~h6 전부 정확히 처리)
        const level = raw.match(/^(#{1,6})\s/)?.[1].length ?? 1
        return { text, level, url: '#' + slugger.slug(text) }
      })
    },
  },
}
```

처음에 TOC 레벨 계산을 이렇게 작성했다가 버그를 만났다.

```ts
// ❌ h4, h5, h6이 전부 level: 1이 되는 버그
const level = raw.startsWith('### ') ? 3 : raw.startsWith('## ') ? 2 : 1
```

`####` 이상의 헤딩은 어떤 조건에도 안 걸려서 전부 level 1이 됐다.
`#`의 개수를 직접 세는 방식으로 고쳤다.

### Blog 문서 타입 정의

```ts
export const Blog = defineDocumentType(() => ({
  name: 'Blog',
  filePathPattern: 'blog/**/*.mdx',
  contentType: 'mdx',
  fields: {
    title: { type: 'string', required: true },   // required: true면 없을 때 빌드 에러
    date: { type: 'date', required: true },
    tags: { type: 'list', of: { type: 'string' }, default: [] },
    lastmod: { type: 'date' },
    draft: { type: 'boolean' },                  // true면 배포에서 제외
    summary: { type: 'string' },
    images: { type: 'json' },
    authors: { type: 'list', of: { type: 'string' } },
    layout: { type: 'string' },
    canonicalUrl: { type: 'string' },
  },
  computedFields,
}))
```

`required: true`로 선언된 필드가 없으면 빌드 타임에 에러가 난다.
이게 contentlayer의 타입 안전성 보장 방식이다.
런타임에 `undefined` 터지는 게 아니라 빌드에서 미리 잡아준다.

### onSuccess 콜백 — 삽질 포인트

빌드 완료 후 검색 인덱스 JSON 파일을 생성하는 콜백이다.

```ts
// ❌ 이렇게 하면 타입 에러
onSuccess: async (importData) => {
  const { allBlogs } = await importData()  // 'allBlogs' 속성이 없다는 에러
}

// ✅ allDocuments에서 type으로 필터링
onSuccess: async (importData) => {
  const { allDocuments } = await importData()
  const allBlogs = allDocuments.filter((doc) => doc.type === 'Blog')
  ...
}
```

contentlayer2에서 `importData()`는 `{ allDocuments: Document[] }` 만 반환한다.
공식 문서나 원본 코드에서 `allBlogs`를 직접 꺼내는 예제를 보고 그대로 했다가
타입 에러를 만났다. `type` 필드로 필터링하는 방식으로 해결했다.

## 4. MDX 파이프라인 — remark와 rehype의 관계

처음에 remark랑 rehype가 뭔지 헷갈렸는데, 변환 파이프라인의 단계별 역할이 다르다.

```
.mdx 파일 (원문)
  ↓
[remark 단계] 마크다운 텍스트 → MDAST (마크다운 AST)
  ├── remark-gfm     : 테이블, 체크박스 등 GFM 문법 파싱
  └── remark-math    : $수식$ 문법 인식

  ↓
[rehype 단계] MDAST → HAST (HTML AST) → HTML 문자열
  ├── rehype-slug              : <h2 id="제목"> 자동 추가
  ├── rehype-autolink-headings : 헤딩에 # 앵커 링크 추가
  ├── rehype-katex             : 수식 → <span class="katex">...</span>
  └── rehype-prism-plus        : 코드 블록에 언어별 색상 클래스 추가
```

플러그인 순서가 중요하다. `rehype-slug`가 `rehype-autolink-headings`보다 먼저 와야 한다.
앵커 링크를 만들려면 먼저 헤딩에 id가 있어야 하기 때문.

## 5. Server Component vs Client Component 경계

Next.js App Router를 쓰면서 이 경계를 제대로 이해해야 한다.

### 문제가 된 코드

처음에 `[slug]/page.tsx`를 이렇게 작성했다.

```tsx
// ❌ Server Component 안에서 Hook 사용
export default function BlogPostPage({ params }) {
  const post = allBlogs.find(...)
  const MDXComponent = useMDXComponent(post.body.code)  // Hook!

  return <MDXComponent />  // 에러: Cannot create components during render
}
```

에러 메시지:

```
Error: Cannot create components during render
Components created during render will reset their state each time they are created.
```

### 원인

`useMDXComponent`는 React Hook이다.
Hook은 `'use client'`로 선언된 Client Component에서만 사용할 수 있다.
App Router에서 기본 컴포넌트는 Server Component이기 때문에 Hook을 쓸 수 없다.

### 해결 — 역할에 따라 분리

```tsx
// src/components/MDXContent.tsx
'use client'  // ← Client Component 선언

import { useMDXComponent } from 'next-contentlayer2/hooks'

export default function MDXContent({ code }: { code: string }) {
  const MDXComponent = useMDXComponent(code)
  return <MDXComponent />
}
```

```tsx
// src/app/blog/[slug]/page.tsx
// Server Component (기본값, 'use client' 없음)
import MDXContent from '@/components/MDXContent'

export default async function BlogPostPage({ params }) {
  const post = allBlogs.find(...)

  return (
    <article>
      <h1>{post.title}</h1>
      <MDXContent code={post.body.code} />  // Client Component에 code만 전달
    </article>
  )
}
```

```
page.tsx (Server Component)
  → 데이터 페칭, SEO 메타데이터, 정적 생성
  → 렌더링 결과가 서버에서 HTML로 생성됨

MDXContent.tsx (Client Component)
  → Hook 사용 (useMDXComponent)
  → 인터랙티브한 MDX 렌더링
```

이 패턴이 App Router의 핵심이다.
"서버에서 할 수 있는 건 서버에서, 클라이언트가 꼭 필요한 부분만 클라이언트에서."

## 6. Next.js 15의 async params

Next.js 15에서 동적 라우트의 `params`가 비동기로 바뀌었다.

```tsx
// ❌ Next.js 14 이하
export default function BlogPostPage({ params }: { params: { slug: string } }) {
  const post = allBlogs.find((p) => p.slug === params.slug)
}

// ✅ Next.js 15
export default async function BlogPostPage({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params  // await 필요
  const post = allBlogs.find((p) => p.slug === slug)
}
```

`generateMetadata`도 동일하게 수정해야 한다.

```tsx
export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const post = allBlogs.find((p) => p.slug === slug)
  if (!post) return {}
  return { title: post.title, description: post.summary }
}
```

## 7. Tailwind 버전 충돌

`create-next-app` 최신 버전으로 생성하면 **Tailwind v4**가 설치된다.
원본 템플릿은 **v3** 기반이고, v4는 문법이 완전히 달라졌다.

```css
/* ❌ v3 문법 — v4에서 동작 안 함 */
@tailwind base;
@tailwind components;
@tailwind utilities;

/* ✅ v4 문법 */
@import "tailwindcss";
```

v4는 `tailwind.config.ts`의 `plugins` 방식도 바뀌고,
`@tailwindcss/typography` 같은 플러그인도 v4 호환 버전이 필요해서 설정이 복잡해진다.

원본 템플릿과 동일한 환경을 맞추려면 v3로 다운그레이드하는 게 낫다.

```bash
npm install tailwindcss@^3.4.14
```

`tailwind.config.ts`에서도 `require()` 대신 ES module import 방식을 써야 ESLint 에러가 안 난다.

```ts
// ❌ require() 방식 - ESLint 에러
plugins: [require('@tailwindcss/typography')]

// ✅ import 방식
import typography from '@tailwindcss/typography'
plugins: [typography]
```

## 8. 다크모드 ThemeSwitch — hydration 이슈

다크모드 토글 버튼을 만들 때 이런 패턴을 쓴다.

```tsx
'use client'

export default function ThemeSwitch() {
  const { theme, setTheme } = useTheme()
  const [mounted, setMounted] = useState(false)

  useEffect(() => setMounted(true), [])
  if (!mounted) return null  // 클라이언트에서만 렌더링

  return (
    <button onClick={() => setTheme(theme === 'dark' ? 'light' : 'dark')}>
      {theme === 'dark' ? '🌞' : '🌙'}
    </button>
  )
}
```

`mounted` 상태를 쓰는 이유가 있다.

서버에서 렌더링할 때는 현재 테마가 뭔지 알 수 없다.
`localStorage`나 `prefers-color-scheme`은 브라우저에서만 접근할 수 있기 때문.

서버에서 렌더링된 HTML과 클라이언트에서 hydration된 결과가 다르면
React가 hydration mismatch 에러를 낸다.

그래서 서버에서는 아무것도 렌더링하지 않고(`return null`),
클라이언트에서 마운트된 후에만 테마에 맞는 버튼을 보여주는 것.

## 9. tsconfig.json — 경로 설정

`src/` 기반 프로젝트에서 자주 헷갈리는 부분이다.

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./*"],
      "contentlayer/generated": ["./.contentlayer/generated"]
    }
  },
  "include": [
    "**/*.ts",
    "**/*.tsx",
    ".contentlayer/generated"
  ]
}
```

`@/*` 경로가 `./*` (루트 기준)면 `data/`는 루트에, `lib/`도 루트에 있어야 한다.
`./src/*`로 설정하면 `src/` 안에서 찾는다.

둘 중 하나로 통일하는 게 중요하다. 섞이면 계속 "모듈을 찾을 수 없습니다" 에러가 난다.

## 마치며

단순히 템플릿을 가져다 쓸 때는 몰랐던 것들을 이번에 직접 만들면서 알게 됐다.

**contentlayer가 하는 일** — MDX 파일을 타입 안전한 JSON으로 변환하고,
빌드 타임에 slug, readingTime, toc 같은 값들을 자동으로 계산해준다.
`allBlogs`를 import하면 완전한 타입이 붙어있는 이유가 이거다.

**Server / Client Component 경계** — App Router에서 기본은 Server Component다.
Hook을 쓰려면 반드시 `'use client'`가 필요하고,
최대한 Server Component로 유지하다가 꼭 필요한 부분만 Client로 분리하는 게 맞는 방향이다.

**버전 관리** — Next.js, Tailwind, contentlayer 각각의 버전이 서로 맞아야 한다.
최신 버전이 항상 좋은 게 아니고, 의존성들이 서로 호환되는 버전 조합을 찾는 게 중요하다.

다음 단계로는 **테스트 세팅(Vitest)**, **Lighthouse CI**, **번들 분석**을 추가해볼 예정이다.
